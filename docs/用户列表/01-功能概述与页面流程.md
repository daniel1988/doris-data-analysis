# 用户列表 - 功能概述与页面流程

本文档基于 `old_frontend` 中的用户列表页面实现，整理该模块的功能目标、路由入口、页面组成与主交互流程，便于后续在新前端中复刻或迁移。

## 1. 模块目标

用户列表页面用于按项目范围查询用户明细，并支持围绕用户属性做筛选、列展示和结果分页。它更像一个轻量的“用户明细检索台”，主要服务于以下场景：

- 查看某项目下的用户明细
- 按时间范围筛选用户
- 按用户 ID 精准检索用户
- 按 A/B 测试分组筛选用户
- 配合动态过滤器做复杂用户属性筛选
- 自定义表格展示列

它不是用户管理后台中的“账号用户列表”，而是业务数据侧的“游戏用户列表”。

## 2. 页面入口

### 2.1 顶部导航入口

在新前端的信息架构中，建议在顶部导航新增一级菜单：

- 用户

并将“用户列表”放在该菜单下作为子菜单：

- 用户
  - 用户列表

这样做的原因是：

- 用户列表、用户标签、用户分群、用户属性天然属于同一组用户运营能力
- 从导航语义上看，“用户”比“游戏用户”更适合作为面向业务方的一级入口
- 后续若继续扩展用户标签、用户分群等页面，也能在该菜单下保持一致层级

如果延续 `old_frontend` 的路由结构，则该菜单实际指向的页面仍然是：

- `/game-users/list`

也就是说，这次补充的是导航分组，不是页面路由重构。

### 2.2 路由入口

用户列表路由定义在：

- `old_frontend/router/modules/gameUsers.ts`

对应路由为：

- `/game-users/list`

路由元信息中的权限为：

- `user_list.view`

### 2.3 页面文件

主页面实现位于：

- `old_frontend/views/game-users/user-list/index.vue`

## 3. 页面结构

用户列表页主要分成 4 个区域：

1. 顶部筛选区
2. 已生效过滤器展示区
3. 表格结果区
4. 两个弹窗：过滤器配置弹窗、列选择弹窗

### 3.1 顶部筛选区

页面默认提供 3 个快捷筛选项：

- 时间范围
- 用户 ID
- A/B 测试

其中：

- 用户 ID 使用 `DimensionValueSelect` 远程选择用户维度值
- A/B 测试也使用 `DimensionValueSelect`，且支持多选

### 3.2 已生效过滤器展示区

如果存在动态过滤器，页面会在筛选表单下方展示过滤器条。

这个区域的作用不是只读展示，而是支持直接修改过滤器值：

- 字段名和操作符以标签形式展示
- 对于需要输入值的操作符，继续嵌入 `DimensionValueSelect`
- 值更新后，会同步回 `activeFilters`

### 3.3 表格结果区

表格区域承担以下职责：

- 展示用户明细数据
- 显示当前总用户数
- 支持翻页和分页大小切换
- 支持字段列选择
- 支持双击复制单元格内容

默认列包括：

- `u_event_time`
- `u_openid`
- `u_ab_test`
- `u_gid`
- `u_channel`
- `u_from_ad_id`
- `u_from_advertiser_id`
- `u_reg_cfg_version`
- `u_reg_game_version`

### 3.4 过滤器配置弹窗

过滤器弹窗复用了：

- `GlobalFilterComponent`

该弹窗负责配置复杂用户属性过滤条件，并在确认后写入 `activeFilters`。

### 3.5 列选择弹窗

列选择弹窗用于决定表格显示哪些字段，支持：

- 搜索字段
- 多选字段
- 查看当前已选字段
- 删除已选字段

## 4. 主流程

### 4.1 进入页面后的初始化流程

页面 `onMounted` 时会执行以下步骤：

1. 从 `localStorage.currentProjectAlias` 读取当前项目别名
2. 调用 `fetchUserProperties()` 拉取当前项目的用户属性元数据
3. 尝试从 `localStorage` 恢复已保存的动态过滤器
4. 执行一次默认查询
5. 监听全局 `gameChanged` 事件，支持项目切换后自动刷新

### 4.2 默认查询行为

页面初始化时会先设置默认时间范围：

- 最近 7 天

随后带着该默认时间范围执行查询，因此用户首次进入页面时通常能直接看到最近一周的数据，而不是空表。

### 4.3 查询主流程

点击“查询”时页面会：

1. 收集顶部快捷筛选条件
2. 收集动态过滤器 `activeFilters`
3. 合并成统一的 `user_filter_group`
4. 按当前表格列生成 `select_fields`
5. 组装分页参数
6. 调用 `userApi.getUserList(payload)`
7. 渲染返回的 `rows` 与 `count`

### 4.4 列选择流程

点击“选择列”时页面会：

1. 打开列选择弹窗
2. 以当前 `columns` 作为初始选中项
3. 用户调整后保存到 `columns`
4. 标记 `userDefinedColumns = true`
5. 重新发起查询

### 4.5 动态过滤器流程

点击“过滤器”时页面会：

1. 将当前 `activeFilters.filters` 深拷贝到 `tempFilters`
2. 打开 `GlobalFilterComponent`
3. 用户调整后点击确定
4. 将结果回写到 `activeFilters`
5. 同步保存到 `localStorage`
6. 自动重新查询

## 5. 项目上下文依赖

用户列表页不是独立选择项目，而是依赖全局项目上下文：

- 页面初始化从 `localStorage.currentProjectAlias` 取值
- 页面运行过程中监听 `gameChanged` 事件

当项目切换时，页面会重新：

- 更新 `projectAlias`
- 拉取用户属性元数据
- 重新查询列表

## 7. 页面特点总结

和用户标签、用户分群等模块相比，用户列表页的特点是：

- 页面结构较轻，但查询逻辑集中
- 依赖项目上下文和用户属性元数据
- 强调“查明细”和“临时筛选”，而不是复杂建模
- 同时承担主页面查询和抽屉下钻复用的设计参考
- 在导航上适合作为“用户”一级菜单下的基础子能力

因此它既是一个独立页面，也是多个业务模块的用户明细展示底座。
