# 用户列表 - 查询与过滤设计

本文档聚焦 `old_frontend/views/game-users/user-list/index.vue` 中的查询、过滤、列选择与持久化设计，说明用户列表页是如何将多个筛选入口统一组装为查询请求的。

## 1. 查询输入来源

用户列表页的查询条件并不是只来自一套表单，而是由三部分共同组成：

1. 顶部快捷筛选项
2. 动态过滤器 `activeFilters`
3. 表格列选择状态

最终所有条件会被合并成一个 `/user_list` 查询 payload。

## 2. 顶部快捷筛选项

页面通过 `filterForm` 维护顶部快捷筛选项：

```ts
const filterForm = reactive({
  timeRange: [] as [Date, Date] | [],
  userId: '' as string | string[],
  abTest: [] as string[]
})
```

### 2.1 时间范围

时间范围使用 `el-date-picker` 的 `daterange` 模式，查询时会转换成：

```ts
{
  column: { table: 'user_data', field: 'u_event_time' },
  operator: 9,
  value: [startISOString, endISOString]
}
```

其中：

- `operator = 9` 表示 Between
- 查询字段落在 `user_data.u_event_time`

### 2.2 用户 ID

用户 ID 使用：

- `DimensionValueSelect`

配置项为：

- `table = 'user_data'`
- `field = 'u_openid'`

查询时会转换为：

```ts
{
  column: { table: 'user_data', field: 'u_openid' },
  operator: 1,
  value: filterForm.userId
}
```

这里相当于提供了一个默认的“精确匹配用户 ID”快捷入口。

### 2.3 A/B 测试

A/B 测试同样使用 `DimensionValueSelect`，但启用了：

- `multiple`
- `clearable`

查询时会转换为：

```ts
{
  column: { table: 'user_data', field: 'u_ab_test' },
  operator: 10,
  value: filterForm.abTest
}
```

即“包含任意一个 A/B 实验值”的筛选方式。

## 3. 动态过滤器

### 3.1 数据结构

动态过滤器存放在：

```ts
const activeFilters = ref<FilterGroup>({ scope: 1, filters: [] })
```

其中：

- `scope = 1` 表示 AND
- `filters` 为具体的过滤条件数组

### 3.2 配置组件

动态过滤器通过弹窗中的：

- `GlobalFilterComponent`

进行编辑。

组件接收：

- `projectAlias`
- `filter-type = 'user'`
- `user-properties = allUserProperties`

因此该页面的动态过滤器，本质上是“基于当前项目用户属性元数据”的通用过滤器编辑器。

### 3.3 生效逻辑

点击“确定”后，页面会：

1. 将 `tempFilters` 深拷贝回 `activeFilters.filters`
2. 固定 `scope = 1`
3. 保存到本地缓存
4. 立即重新查询

这意味着旧实现里动态过滤器组默认只支持：

- AND 组合

## 4. 动态过滤器的有效值过滤

在真正查询前，页面不会直接把 `activeFilters.filters` 原样发给接口，而是会先过滤掉无效条件。

主要规则如下：

### 4.1 不需要值的操作符

对于以下操作符，允许没有 `value`：

- `IS_NULL`
- `IS_NOT_NULL`

这类操作符在旧代码中由 `needsValue()` 判断。

### 4.2 需要值的操作符

若操作符需要值，则会排除以下无效情况：

- `null`
- `undefined`
- 空字符串
- 空数组
- 数组内全部为空值

这样做的目的是避免把“未填完整的过滤条件”带进请求体，导致接口报错或查出异常结果。

## 5. 已生效过滤器展示区

页面会将 `activeFilters.filters` 以紧凑条目的形式展示在筛选区域下方。

每条过滤器展示内容包括：

- 字段中文名
- 操作符中文名
- 当前值

这一区域的特点是：

- 它不是纯展示
- 对需要值的过滤器，会继续渲染 `DimensionValueSelect`
- 改值后会直接调用 `updateFilterValue()` 回写到 `activeFilters`

这种设计让用户可以在不重新打开弹窗的情况下快速调整某个过滤值。

## 6. 操作符显示与占位文案

页面内置了 `operatorLabelMap`，用于将数字操作符映射为中文文案，例如：

- `1` -> 等于
- `5` -> 大于
- `9` -> 在..之间
- `10` -> 包含
- `13` -> 正则匹配

同时还通过 `getValuePlaceholder()` 为不同操作符生成不同占位提示。

例如：

- `9` 提示输入范围值
- `10` / `11` 提示多值选择语义
- `13` 提示正则表达式格式

## 7. 列选择逻辑

### 7.1 默认列

页面初始化时内置一组默认列，作为首屏展示字段。

同时会记录：

```ts
const initialDefaultColumns = ref<string[]>(columns.value.map(c => c.prop))
```

这个值用于后续列选择时，维持“默认字段优先、用户追加字段靠后”的顺序。

### 7.2 临时选中列

列选择弹窗使用：

```ts
const tempSelectedColumns = ref<string[]>([])
```

保存时会根据当前选中列重新构建 `columns`。

### 7.3 保存顺序规则

`handleColumnSave()` 会按如下规则排序：

1. 先保留默认字段中仍被选中的部分
2. 再追加用户后来新增的字段

因此旧实现并不是简单按勾选顺序展示，而是刻意维持一套稳定顺序，避免默认核心列被打散。

### 7.4 自定义列标记

保存列选择后，页面会设置：

```ts
userDefinedColumns.value = true
```

一旦用户手动改过列，就不再轻易被服务端返回的列配置覆盖。

## 8. 与服务端返回列的关系

在查询成功后，如果：

- `userDefinedColumns.value === false`

则页面会尝试用服务端返回的 `columns` 覆盖表格列。

逻辑优先级大致是：

1. 读取 `data.columns`
2. 若没有，再尝试从首行 `rows[0]` 推断字段
3. 最多取前 12 列

这套逻辑说明旧接口具备“服务端下发推荐列”的能力，但最终仍然允许用户自行接管列展示。

## 9. 过滤器本地持久化

### 9.1 存储键

页面使用固定 localStorage 键：

- `userList_filters`

### 9.2 保存内容

保存结构包括：

- `scope`
- `filters`
- `timestamp`

### 9.3 恢复时机

页面初始化时会先执行：

- `loadFiltersFromStorage()`

恢复成功后，再触发查询。

因此用户离开页面再回来时，通常可以继续沿用上一次的过滤器条件。

### 9.4 清理策略

旧实现中存在 `clearFiltersFromStorage()` 方法，但主流程没有显式调用，说明当前更偏向“长期保留过滤条件”，而不是离开即清空。

## 10. 结果表格行为

表格除了展示数据，还有两个交互细节：

### 10.1 双击复制

- `u_openid` 列支持双击复制
- 其他普通单元格也支持双击复制

复制逻辑优先使用：

- `navigator.clipboard.writeText`

若失败，再回退到传统 `textarea + execCommand('copy')`。

### 10.2 分页

分页逻辑由：

- `pageNum`
- `pageSize`
- `total`

控制。翻页或修改每页条数后，页面会立即重新查询。

## 11. 设计总结

用户列表页的筛选设计可以概括为：

- 用少量快捷条件覆盖高频需求
- 用通用过滤器承接复杂条件
- 用本地持久化保留用户上下文
- 用列选择满足结果展示灵活性

因此它虽然看起来只是一个“列表页”，本质上已经具备了一个轻量查询工作台的能力。
