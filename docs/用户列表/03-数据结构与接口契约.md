# 用户列表 - 数据结构与接口契约

本文档整理 `old_frontend` 用户列表页面依赖的数据结构和接口协议，重点说明 `/user_list` 查询接口的请求体、返回结构以及页面对字段的使用方式。

## 1. 接口定义位置

用户列表相关接口定义在：

- `old_frontend/api/user.ts`

其中和业务用户列表页面直接相关的方法主要有：

- `getUserList(payload)`
- `getUserProperties(projectAlias)`

## 2. 查询请求体结构

`getUserList` 使用的请求体类型为：

```ts
export interface UserListPayload {
  project_alias: string
  select_fields?: string[]
  user_filter_group?: FilterGroup
  event_filter_group?: FilterGroup
  tag_filters?: Array<{
    tag_code: string
    operator: number
    tag_value?: any
  }>
  user_group_filters?: Array<{
    group_name?: string
    group_code: string
    operator: number
  }>
}
```

虽然主页面 `views/game-users/user-list/index.vue` 主要使用：

- `project_alias`
- `select_fields`
- `user_filter_group`

但从接口定义可以看出，用户列表查询底层其实预留了更丰富的过滤能力：

- 事件过滤
- 标签过滤
- 用户分群过滤

这也是它能被抽屉下钻组件复用的基础。

## 3. 过滤器结构

### 3.1 `FilterGroup`

```ts
export interface FilterGroup {
  scope: 0 | 1
  filters: Filter[]
}
```

其中：

- `scope = 0` 表示 OR
- `scope = 1` 表示 AND

### 3.2 `Filter`

```ts
export interface Filter {
  column: {
    table: string
    field: string
    alias?: string
  }
  operator: number
  value?: any
}
```

单个过滤器由三部分组成：

- 字段定位 `column`
- 操作符 `operator`
- 对应值 `value`

这套结构和分析模块、分群模块中的过滤器协议高度一致，说明它们共享同一套后端过滤语言。

## 4. 主页面实际使用的 payload

`views/game-users/user-list/index.vue` 查询时实际组装的 payload 大致如下：

```ts
{
  project_alias: projectAlias.value,
  select_fields: columns.value.map(c => c.prop),
  page_size: pageSize.value,
  page_num: pageNum.value,
  user_filter_group: {
    scope: activeFilters.value.scope || 1,
    filters: final_filters
  }
}
```

注意这里的 `page_size` 和 `page_num` 并没有显式出现在 `UserListPayload` 类型中，说明：

- 旧前端实际请求体比接口类型定义更宽
- `UserListPayload` 没有完整描述当前页面使用的全部字段

后续若迁移到新前端，建议补全该类型，避免“实际可传字段”和“类型声明”不一致。

## 5. 主页面过滤字段来源

### 5.1 时间范围字段

时间范围过滤字段：

- `user_data.u_event_time`

### 5.2 用户 ID 字段

用户 ID 过滤字段：

- `user_data.u_openid`

### 5.3 A/B 测试字段

A/B 测试过滤字段：

- `user_data.u_ab_test`

### 5.4 表格默认展示字段

默认列主要也是用户属性字段，例如：

- `u_event_time`
- `u_openid`
- `u_gid`
- `u_channel`
- `u_reg_cfg_version`

因此这个页面从底层上看，是一个“以 `user_data` 表为中心”的用户属性查询页。

## 6. 查询返回结构

主页面查询成功后，会按如下方式读取返回值：

```ts
const data = res?.data || {}
const rows = Array.isArray(data?.rows) ? data.rows : []
const columnsRaw = Array.isArray(data?.columns) ? data.columns : []
const count = Number(data?.count || rows.length || 0)
```

由此可以推断 `/user_list` 返回结构至少包含：

```ts
{
  code: number,
  message?: string,
  data: {
    rows: any[],
    columns?: any[],
    count?: number
  }
}
```

其中：

- `rows` 为表格行数据
- `columns` 为服务端建议展示列
- `count` 为总记录数

页面还要求：

- `code === 0` 才视为成功

这与部分后台管理接口使用的 `200` 语义不同，说明该接口更接近 Doris 查询类接口的返回风格。

## 7. 服务端列配置的兼容逻辑

查询返回后，页面在 `userDefinedColumns === false` 时，会尝试按以下顺序确定表格列：

1. 使用 `data.columns`
2. 若缺失，则使用首行 `rows[0]` 的字段名

然后再转成：

```ts
columns.value = serverColumns.slice(0, 12).map(k => ({
  prop: k,
  label: getPropertyName(k)
}))
```

因此服务端返回的列定义不是强制性的，而是“在用户没有自定义列时的推荐展示方案”。

## 8. 用户属性元数据接口

页面还会调用：

```ts
userApi.getUserProperties(projectAlias)
```

对应请求地址为：

- `/event_analysis/user_properties`

参数为：

- `project_alias`

页面对该接口返回值做了兼容处理：

```ts
const props = Array.isArray(res)
  ? res
  : Array.isArray(res.data)
    ? res.data
    : Array.isArray(res.data?.data)
      ? res.data.data
      : []
```

说明旧系统中该接口也存在返回层级不稳定的问题。

该接口的主要用途是：

- 提供过滤器字段列表
- 提供列选择字段列表中的中文名和字段名

## 9. 字段中文名映射

为了保证即使元数据未返回，页面也能正常展示，主页面内置了一份 `fieldLabelMap`，例如：

- `u_event_time` -> 注册时间
- `u_openid` -> 用户ID
- `u_ab_test` -> A/B测试
- `u_gid` -> 游戏ID
- `u_channel` -> 渠道

字段展示名的优先级是：

1. 优先使用元数据接口返回的 `propertyName`
2. 若没有，则回退使用本地 `fieldLabelMap`
3. 再没有，才直接展示原始字段名

## 10. 抽屉下钻复用时的扩展过滤能力

虽然主页面本身只用到 `user_filter_group`，但在 `UserListContent.vue` 中，这套接口还能继续承载：

- `tag_filters`
- `user_group_filters`

这说明 `/user_list` 被设计成一个可复用的统一用户明细查询接口，而不是只服务当前页面。

## 11. 协议层注意事项

迁移时需要特别注意以下几点：

1. `/user_list` 的请求体实际字段比 `UserListPayload` 类型定义更多。
2. 返回码使用 `code === 0` 表示成功，不是常见的 `200`。
3. `getUserProperties` 和 `/user_list` 都存在多层返回结构兼容逻辑。
4. 列配置既可能来自前端默认值，也可能来自服务端返回值，还可能来自用户自定义选择。

这些兼容逻辑如果不在文档中明确，重构时很容易漏掉边界行为。
