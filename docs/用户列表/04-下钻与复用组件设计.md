# 用户列表 - 下钻与复用组件设计

本文档整理 `old_frontend` 中和用户列表相关的抽屉下钻与复用组件设计，说明为什么用户列表不仅是一个独立页面，也是一套可嵌入其他模块的用户明细展示能力。

## 1. 复用组件入口

和用户列表直接相关的两个核心复用组件为：

- `old_frontend/components/common/UserDrilldownDrawer.vue`
- `old_frontend/components/common/UserListContent.vue`

它们一起组成了“抽屉式用户明细下钻”能力。

## 2. `UserDrilldownDrawer` 的职责

### 2.1 组件定位

`UserDrilldownDrawer.vue` 是一个外层抽屉容器，负责：

- 控制抽屉开关
- 根据下钻模式设置标题
- 将标签过滤或分群过滤转换为子组件入参
- 接收实际总数并动态更新标题

### 2.2 支持的模式

组件通过 `mode` 区分两类下钻：

- `tag`
- `group`

当：

- `mode = 'tag'` 时，渲染 `UserListContent` 并传入 `initialTagFilter`
- `mode = 'group'` 时，渲染 `UserListContent` 并传入 `initialUserGroupFilter`

这说明旧系统把“标签命中用户”和“分群命中用户”都统一映射成同一个用户明细展示组件。

### 2.3 标题逻辑

抽屉标题不是固定字符串，而是根据：

- 标签名或分群名
- 初始用户数
- 查询后真实总数

动态计算得出。

例如在标签模式下：

- 初始显示标签名称和估算用户数
- 如果子组件查询出了更准确的 `total`，则用真实总数更新标题

## 3. `UserListContent` 的职责

### 3.1 组件定位

`UserListContent.vue` 可以理解为“可嵌入的用户列表内容区”，它复用了主页面的大部分展示能力，但输入条件来自外部 props，而不是顶部表单。

它主要负责：

- 根据初始标签过滤或分群过滤查询用户明细
- 展示当前命中的过滤条件
- 渲染用户表格
- 支持列选择
- 支持导出
- 向父组件回传总用户数

### 3.2 输入参数

组件支持的 props 包括：

- `initialTagFilter`
- `initialUserGroupFilter`
- `projectAlias`

其中：

- `initialTagFilter` 包含 `tagCode`、`tagShowName` 等信息
- `initialUserGroupFilter` 包含 `groupCode`、`groupName`、`operator` 等信息

### 3.3 输出事件

组件会通过：

- `totalChange`

将当前查询到的真实总数通知给外层抽屉。

## 4. 标签下钻链路

用户标签列表页中，点击用户数后会打开：

- `UserDrilldownDrawer`

并传入：

```ts
{
  id,
  tagCode,
  tagShowName,
  userCount,
  projectAlias
}
```

随后链路为：

1. 标签列表页设置 `selectedTag`
2. 打开 `drawerVisible`
3. `UserDrilldownDrawer` 根据 `mode='tag'` 计算入参
4. `UserListContent` 按标签过滤查询用户明细
5. 结果总数通过 `totalChange` 回传
6. 抽屉标题更新为“标签名 + 实际人数”

## 5. 分群下钻链路

虽然当前需求聚焦用户列表页面，但从组件设计上看，用户分群页也复用了相同的下钻抽屉。

这说明旧前端刻意把“查看命中的用户明细”抽象成了统一能力，而不是分别在标签页和分群页各写一套表格。

## 6. `UserListContent` 的过滤策略

### 6.1 标签过滤

在标签模式下，组件会把外部传入的标签过滤转换为：

- `tag_filters`

并附带到 `/user_list` 请求体中。

这与主页面直接使用 `user_filter_group` 不同，说明同一个查询接口承载了多种入口来源：

- 页面表单过滤
- 标签命中用户过滤
- 分群命中用户过滤

### 6.2 分群过滤

在分群模式下，组件会把分群信息转换为：

- `user_group_filters`

并附带到查询请求中。

因此：

- 主页面是“直接查询用户属性”
- 抽屉内容页是“带业务上下文地查询用户明细”

## 7. 展示层复用

`UserListContent` 在展示层延续了主页面的大部分能力：

- 表格列渲染
- 时间字段格式化
- 双击复制
- 列选择弹窗
- 分页

这类设计的优点是：

- 主页面和抽屉体验一致
- 业务模块只需关心入口过滤条件，不需要重复实现表格

但也意味着：

- 用户列表展示逻辑被复制了两份思路
- 主页面和复用组件之间并没有完全共享一个 composable 或通用表格容器

## 8. 和主页面的关系

可以把这三层关系理解为：

1. `views/game-users/user-list/index.vue`
   - 完整页面版用户列表
2. `components/common/UserListContent.vue`
   - 可嵌入版用户列表内容
3. `components/common/UserDrilldownDrawer.vue`
   - 下钻容器层

这种结构体现了旧前端的一种折中方案：

- 不完全抽成统一页面组件
- 但把“下钻明细”单独抽成可复用能力

## 9. 与顶部导航的关系

如果在新前端顶部导航中新增一级菜单：

- 用户

并把“用户列表”作为其子菜单，那么这套复用能力的定位会更清晰：

- 用户列表
  - 作为“用户”菜单下的基础查询页面
- 用户标签 / 用户分群
  - 作为“用户”菜单下的衍生业务能力
- `UserDrilldownDrawer` / `UserListContent`
  - 作为这些业务能力共享的用户明细底座

也就是说，导航层的“用户 > 用户列表”是入口，而抽屉与内容组件是支撑其他用户业务页面继续下钻的通用能力。

## 10. 设计价值

这套抽屉下钻设计有两个明显价值：

### 10.1 提升复用性

标签页、分群页都可以直接复用同一个用户明细抽屉，而不需要重复写查询表格。

### 10.2 保持上下文连续性

用户不需要离开当前业务页面，就能在右侧抽屉内查看命中的用户明细，这比跳转到单独页面更连贯。

## 11. 迁移建议

如果未来在新前端重构用户列表相关能力，建议优先考虑以下方向：

1. 将主页面和 `UserListContent` 的公共查询逻辑抽成 composable。
2. 将表格列、复制、分页、格式化逻辑抽成共享表格容器。
3. 保留 `UserDrilldownDrawer` 这种“业务页内下钻”模式，因为它的交互价值很高。
4. 统一主页面查询和下钻查询的 payload 构建层，避免页面内散落转换逻辑。
5. 在信息架构上将用户列表放入顶部“用户”菜单下，强化它作为用户业务基础入口的角色。

这样既能保留旧体验的优点，也能减少重复实现成本。
