# AI 数据分析（Chat2SQL）开发实施计划

基于前期对于元数据上下文组装、大模型接入、前端可视化的设计，本实施计划详细阐述了 Chat2SQL 功能在代码工程级别的落地细节。前后端模块严格保持独立，以便于后续模块的维护与扩展。

## 一、 后端安全与 AI 处理模块

在 Go 后端服务的 `analytics` 业务网关下构建独立的通信服务，包含完整的业务拦截与查询能力：

### 1. 核心 AI 处理服务
**[新建文件] `backend/internal/service/analytics/ai_chat_service.go`**
本服务封装从接收文字到返回执行结果的全链路编排。包括以下核心方法：
- `BuildVirtualSchema(projectAlias string)`: 后端向 `dmp_center` 查询该项目的核心行为事件、已挂载的属性以及基础画像信息，组装成给大模型“阅读”的 Markdown schema (Virtual Schema)。
- `CallLLMForSQL(sysPrompt, userQuery string)`: 通过标准的 HTTP 请求将系统级 Prompt 发往第三方大语言模型 (兼容 OpenAI API 的如 DeepSeek/Qwen 等)，并接收封装了 SQL、`viz_type` 和文字描述的 JSON 回复。
- `ValidateAndSecureSQL(sql, projectAlias string)`: **安全性重中之重**。通过轻量级 AST 解析器 (如 `sqlparser` 或 `vitess`) 解析 LLM 返回的代码树：
  1. 拦截非 `SELECT` 的任何 SQL 动词。
  2. 验证访问表名是否在 `[event_data, user_data, user_tag_data]` 的白名单内。
  3. **数据围栏 RLS**：强制通过 AST 写入 `WHERE project_alias = '...'` 查询条件。
  4. 注入查询保护（如缺少聚合时强制 `LIMIT 1000`）。
- `HandleChat(projectAlias, userQuery)`: 主编排器，当发现验证通过的 SQL 在 Doris 层抛出查询语法错误时，进行报错信息的捕获并打回给 LLM 进行自动修复 (最大重试上限 2 次)。

### 2. HTTP 控制器接入
**[新建文件] `backend/internal/api/ai_chat.go`**
处理与前端接口侧的直接交互。
- 暴露 `/analytics/ai/chat` API，接收 JSON (含用户问题提问 Text)。
- 从会话或请求头中解析 Token 并抽取 `projectAlias`，安全地把参数流转到服务层 `AIChatService`。

### 3. 路由注册
**[修改文件] `backend/internal/api/router.go`**
- 引入该路由使其在现有接口网关下生效。

## 二、 前端独立可视化分析页面

前端将新增一整套用于呈现极具现代化审美的可交互对话面板应用：

### 1. 智能分析主控台
**[新建文件] `frontend/src/views/analytics/ai/index.vue`**
此页面建立左右分栏式 UI (或上下流式 UI)：
- **对话面板**：呈现用户与助手的对话气泡。
- **智能排版系统**：
  - 对文本指令和模型思考过程进行常规气泡渲染。
  - 对接收到的分析产生的 SQL 进行高亮代码展示。
  - 解析后端传来的 `viz_type`, `x_axis`, `y_axis` 的元提示指令，并直接注入 `Echarts` 实例渲染图表 (如折线图、饼形图)。
  - `Fallback` 机制：当 AI 指定的维度复杂导致图表渲染失败时，或者未推荐图表时，无缝降级为展现一个分页的 `ElTable` 显示基础二维表数据。

### 2. 菜单路由集成
**[修改文件] `frontend/src/router/` 相关文件** (如 `router.ts`)
- 配置异步路由 `/analytics/ai` 加载 `ai/index.vue` ，使该页面融入分析业务主菜单当中。
