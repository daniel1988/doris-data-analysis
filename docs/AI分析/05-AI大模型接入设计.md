# AI大模型接入设计

在 Chat2SQL（对话式数据分析）架构中，大语言模型（LLM）扮演着“数据翻译官”的核心角色。本篇档将详细说明模型选型建议、对接协议架构（HTTP REST vs MCP）、Prompt 组装规则以及在后端的通信高可用设计。

## 1. 架构协议路线：REST API 还是 MCP？

目前绝大多数的 LLM 对接有两种核心协议方案。针对本系统的“前端UI -> 后端网关 -> LLM -> Doris”数据流，我们建议**首选兼容 OpenAI 规范的 REST API 模式，无需强依赖 MCP**。

| 维 度 | 方案 A：REST API + 自建网关（**推荐**） | 方案 B：MCP (Model Context Protocol) 架构 |
| -- | -- | -- |
| **交互逻辑** | 后端主动查 Doris 元数据 $\rightarrow$ 拼接拼图(Prompt) $\rightarrow$ 向 LLM 发送对话请求 $\rightarrow$ 收到 SQL $\rightarrow$ AST 沙箱拦截 $\rightarrow$ 查库 | 后端暴露出标准的 MCP Tools（如 `get_schema`, `execute_sql`），由外部的 MCP Agent 主动发起调度。 |
| **安全控制** | 及高。前端仅发送自然语言，后端全权决定注入什么 Context，并在拿到 SQL 后强制过 AST 拦截引擎，实现无缝的多租户（RLS）控制。 | 适中。若允许 LLM 自主决定调用 `execute_sql` 工具，那么沙箱和越权阻断防御的面会复杂很多。 |
| **适用场景** | 构建一个面向上位网页用户、高度客制化的 SaaS 数据后台（本系统目标场景）。 | 构建可以被 Claude Desktop、Cursor 等通用AI工具随时挂载访问的数据平台。 |

> **结论**：本模块的开发**不需要**专门构建 MCP 服务。您只需在 Go 后端引入常见的 HTTP 请求机制即可（由于主流开源及闭源大模型对 OpenAI 接口的高内聚兼容，您甚至可以轻松使用如 `go-openai` 等 SDK 统一对接）。

## 2. 大模型能力选型建议

要能根据提供的 Virtual Schema 生成准确的多表长句 SQL 以及使用特定的函数（如 Doris 的特定时间函数），模型必须具有极强的**逻辑推理、代码生成 (Coding) 与长上下文关注机制**。

推荐接入以下厂商的模型：
1. **DeepSeek (推荐)**：特别是 `DeepSeek-Coder` 或 `DeepSeek-V3/R1`，其代码生成能力极强，且通常天然兼容 OpenAI 接口协议，API 成本极具性价比。
2. **通义千问 (Qwen-Max 或 Qwen-Coder)**：针对中文环境的数据结构具有很好的理解力，且在阿里云上有非常成熟的调用体系支持。
3. **OpenAI (GPT-4o)**：行业标杆，SQL 零样本生成准确率最高，可作为兜底保障或高级付费用户的选项。

*(所有上述模型均支持标准 `Chat Completion` API 入参体系。)*

## 3. 标准的 Prompt 组装结构

每一次向 LLM 发起的请求报文中（`messages` 数组），必须包含且不仅限于以下层级构造：

```json
[
  {
    "role": "system",
    "content": "You are a senior Apache Doris data analyst...\n\n[此处注入 02-元数据上下文设计 篇生成的 Virtual Schema 与可用列]\n\nCRITICAL RULES:\n1. Only return valid SELECT SQL.\n2. Do NOT use markdown code blocks, just raw SQL.\n3. User project scope is isolated, strictly follow the rules."
  },
  {
    "role": "user",
    "content": "帮我看看上个月等级大于20的用户，充值金额排名前十的是哪些人？"
  }
]
```
*注：为了降低 Token 成本并提速，`system` 角色下的 Context 内容是通过后端根据当前提问（或所属 Project）去 `dmp_center` 动态拼装的，而非给出一份全库 DDL。*

## 4. 后端请求的稳健性设计 (Stability & Fallback)

调用外部第三方 AI 服务极其容易遇到网络波动、模型超载（Rate Limit）或响应超时。因此后端的 AI Gateway 节点需要实现：

1. **超时与自动重试**：建议设定单次 HTTP 请求超时界限为 `30s-45s`。超过抛错并引入指数退避（Exponential Backoff）重试，最多重试 2 次。
2. **熔断器预警**：一旦第三方 API 节点连续出现如 429 (Too Many Requests) 或 500 错误，需要触发告警（通知运维）并可快速切换冗余备用模型（如主调 DeepSeek 挂了，系统配置中可以自动 Failover 到同生态的 Qwen 或 GPT）。
3. **流式响应拦截控制 (SSE 兼容)**：
   如果是普通对话，我们可能会用 Stream，但**生成 SQL 供后端执行不推荐单纯流式输出给前端**。
   最佳交互为：流式收取生成的 SQL（或让模型以 JSON 隔离思考过程与 SQL 结果），后端在内部完整提取出 SQL 字符串后，立即丢给 **AST Sandbox (拦截沙箱)**，校验通过后再去 Doris 跑数据，最后将数据和最终修饰过的话术推给前端。
