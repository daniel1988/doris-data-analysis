# 元数据上下文设计 (Context Builder)

大模型生成准确 SQL 的前提，是给予它**准确、精简且带标注的表结构（Schema）和枚举（Enum）** 字典。直接把 Doris 全库的 DDL 给 LLM 效果极差，且会造成 Token 严重浪费。

得益于我们的元数据原生存储于 **Doris `dmp_center` 库**中，我们可以通过直接查询元数据表，为 LLM 动态拼接高度定制化的上下文。

## 1. 动态 Prompt 组装策略 (基于 dmp_center 查询)

我们需要在用户发起请求时，针对其所属的 **Project (项目别名 `project_alias`)**，查询相应的实体定义。

### 1.1 背景：元数据表结构映射

我们的底层业务核心宽表虽然抽象为 `event_data` 和 `user_data`，但为了让大模型准确理解里面的自定义字段并生成合法 SQL，我们将它们的「元数据」托管于 `dmp_center` 库进行系统级闭环管理：

- **`event_data` 的元数据管理**:
  - `project_event`: 管理项目中所有合法的事件（支持 `e_event_name` 过滤验证）。
  - `project_property`: 管理所有的事件属性定义（数据采集服务会自动将这些自定义字典解析为 `event_data` 表的独立物理列）。
  - `project_event_property`: 重中之重，它维护着各个事件**具体绑定了哪些属性**。通过该表联动查询，我们可以给大模型提供“剪裁”后的精准表结构上下文（仅展示当前提问事件相关的物理列），大幅降低 Token 的浪费并减少幻觉。

- **`user_data` 的元数据管理**:
  - `user_properties`: 管理用户实体的属性字典（同样会被数据采集服务自动解析为 `user_data` 表的独立物理列）。
  - `user_tags`: 管理经过每日构建产生的用户标签池。

### 1.2 动态获取元数据字典构建 Schema

当用户发起自然语言对话时，后端服务将动态抓取特定项目（Project）的上述实体以构建 Prompt（基于 `project_alias = '?'`）。

**1. 提取事件列表：**
```sql
SELECT event_id, event_name FROM dmp_center.project_event WHERE project_alias = '?';
```

**2. 提取相关事件的专属属性（联动过滤）：**
相比于全量查询，更好的做法是先通过提问词提取目标事件，然后查询此事件关联的属性字典。
```sql
SELECT p.property_id, p.property_name, p.data_type 
FROM dmp_center.project_property p
JOIN dmp_center.project_event_property ep ON p.property_id = ep.property_id
WHERE p.project_alias = '?' AND ep.event_id IN ('?');
```

**3. 提取用户的基础属性：**
```sql
SELECT property_id, property_name, data_type FROM dmp_center.user_properties WHERE project_alias = '?';
```

最后，服务端将这些查询结果拼接，生成供给 LLM 的 Virtual Schema 背景：
```markdown
You are an expert Apache Doris SQL data analyst.
The user wants to query data from a gaming project '{project_alias}'.
Here are the tables you have access to:

TABLE `event_data` (User behavior events)
- `e_openid` (String): 用户唯一ID
- `e_event_name` (String): 事件标识，取值如: "login", "pay", "level_up"
- `e_event_time` (Datetime): 事件发生时间
  当前业务所涉及的自定义属性 (已通过关联表精简并自动展平为列): 
- `amount` (double): 支付金额
- `item_id` (varchar): 道具ID

TABLE `user_data` (User profiles)
- `u_openid` (String): 用户唯一ID
- `u_event_time` (Datetime): 用户进入系统时间
  当前用户的自定义属性 (已自动展平为列):
- `channel` (varchar): 注册渠道
- `vip_level` (int): VIP等级
```

## 2. 维度枚举过滤与 RAG (检索增强)
有些业务维度包含长尾的字典值（例如千余种商品 ID、数百个渠道代码），通过 `dmp_center.user_tags` 查阅标签定义。
如果在用户对话时提到了特定渠道或标签（如：“买过复活币的大R用户”），我们可以通过轻量的倒排词典或 RAG 向量匹配，将相似度最高的属性名或 `tag_code` 补充进 Prompt。
