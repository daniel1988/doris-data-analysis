# 09-AI对话与结果引用模式设计

## 1. 需求背景与概述
当前的 AI 分析模块主要采用 **Chat2SQL** 模式，即将用户的自然语言意图转换为 SQL 并执行，最后将结果通过可视化图表展示给用户。
为了增强 AI 助手的灵活性与交互深度，计划在现有的 AI 分析界面中引入 **多模式切换** 功能：
1. **智能取数 (Chat2SQL)**：即现有的自然语言生成 SQL 并查询数据的模式。
2. **AI 对话 (Chat with AI)**：直接与 AI 进行自然语言交互的模式，主要用于数据解读、思路扩展、业务咨询等。

在 **AI 对话** 模式下，系统需要支持 **“引用上下文/返回结果”** 的能力。用户可以选择将前面 Chat2SQL 执行得到的图表数据或表格数据作为上下文，发送给大模型进行进一步的深度分析与解读。

## 2. 核心功能设计

### 2.1 交互模式切换
* 在 AI 对话输入框（`ChatInput.vue`）的上方或左侧，增加一个模式切换开关（如 `Radio Button` 或 `Tabs`），包含 `[智能取数]` 和 `[AI对话]` 两个选项。
* **智能取数模式**：保持现有逻辑，输入框主要接收取数需求。
* **AI 对话模式**：输入框接收任意问题。此时，若当前界面存在已经执行成功的查询结果（如表格、图表），输入框上方提供一个 `[✓ 引用当前结果数据]` 的可勾选项。

### 2.2 数据引用机制
* **引用的内容**：当勾选“引用结果”时，前端在发送请求时，需要提取当前最新的（或用户指定关联的）查询结果集（JSON格式），以及对应的表头信息。
* **数据截断策略**：考虑到 LLM 的上下文窗口限制（Token 数量限制），若返回的行数过多（如超过 50 行），系统需要自动截断数据，仅将前 N 行作为样本数据连同统计信息（如总行数、字段名）发送给模型，并在 Prompt 中明确说明“这是部分样本数据”。

### 2.3 消息展示分离
* 在消息列表组件（`ChatMessage.vue` / `index.vue`）中，需增加对消息类型的区分。
* 若为 `Chat2SQL` 模式，展示逻辑不变（包含 SQL 生成状态、图表渲染等）。
* 若为 `Chat` 模式，则直接以 Markdown 格式渲染 AI 的纯文本回复，不需要走 SQL 解析与 ECharts 渲染流程。

## 3. 前端改造方案

### 3.1 状态管理与接口调整
* 扩展前端发送消息的结构，增加 `mode` 和 `context_data` 字段：
  ```typescript
  interface ChatMessagePayload {
    project_alias: string;
    message: string;
    model_id: number;
    mode: 'sql' | 'chat';      // 模式标识
    context_data?: any;        // 当 mode 为 chat 时，携带引用的 JSON 结果数据
  }
  ```

### 3.2 UI 组件改造
1. **`ChatInput.vue`**：
   * 增加模式切换控件（Radio / Segmented 控件）。
   * 监听模式切换，在 `chat` 模式下如果检测到当前上下文中存在结果数据，显示“引用数据”的 Checkbox 或 Tag。
2. **`ChatMessage.vue`**：
   * 扩展气泡渲染逻辑。对于 `mode === 'chat'` 的返回，直接渲染为 Markdown（引入 `markdown-it` 或类似库），并隐藏执行 SQL 的 Action Footer（如“保存为有效分析”可改为仅保存文本对话，或禁用）。

## 4. 后端改造方案

### 4.1 接口扩展
* **API 路由**：复用现有的 `/api/v1/analytics/ai/chat` 接口，但根据 Payload 中的 `mode` 走不同的 Service 逻辑。
* **Request DTO**：
  ```go
  type AIChatRequest struct {
      ProjectAlias string      `json:"project_alias" binding:"required"`
      Message      string      `json:"message" binding:"required"`
      ModelId      int         `json:"model_id"`
      Mode         string      `json:"mode"`         // "sql" | "chat"
      ContextData  interface{} `json:"context_data"` // 供 chat 模式引用的数据
  }
  ```

### 4.2 核心服务改造 (`ai_chat_service.go`)
增加一个新的方法 `ChatWithAI`，绕过 Virtual Schema 生成和 SQL 执行流程。
* **System Prompt 组装**：
  如果用户传递了 `ContextData`，后端需要构建如下形式的 Prompt 注入给大模型：
  ```text
  你是一个专业的数据分析师。用户正在查看以下数据（JSON格式）：
  {ContextData 序列化后的字符串，注意限制长度}

  请根据上述数据和用户的提问进行分析解读。
  ```
* **LLM 调用与返回**：
  直接将用户问题与 System Prompt 发送给 LLM，获取回答后，将 `content` 作为纯文本返回给前端。响应的 `type` 可标识为 `text`。

## 5. 实施步骤建议
1. **后端接口适配**：在 `ai_chat_service.go` 中解析 `mode`，并添加纯对话分支的逻辑实现。支持 `ContextData` 的接收与 Prompt 拼接。
2. **前端 UI 结构调整**：在 `ChatInput.vue` 中添加模式切换按钮及“引用当前数据”的勾选项。
3. **前端渲染适配**：更新 `ChatMessage.vue` 以支持 Markdown 格式的纯文本对话渲染。
4. **全链路联调测试**：先执行一次 Chat2SQL 获取数据，然后切换到 AI 对话模式，勾选数据并提问（如“帮我分析一下这些数据体现的趋势”），验证 AI 的回答是否结合了上下文数据。
