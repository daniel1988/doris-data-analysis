# 07-原子层重构：过滤操作符选择器设计

## 1. 概述
在过滤器（Filter）组件中，操作符（Operator）起着连接“字段”和“值”的桥梁作用。不同的字段数据类型（如数字、字符串、日期）所支持的操作符是不同的（例如，字符串不支持“大于/小于”，数字不支持“包含字符串”等）。
**过滤操作符选择器（Operator Selector）** 的核心职责是：根据当前选中的字段类型，动态渲染合法且合理的操作符下拉列表。

## 2. 现有实现分析（基于 `old_frontend`）

### 2.1 原逻辑回顾
在老版的 `GlobalFilterComponent.vue` 或类似的过滤器组件中，操作符的加载主要依赖于一个名为 `getOperatorOptions(f)` 的方法。
其核心逻辑如下：
1. **获取数据类型**：从字段元数据（如 `_dataType` 或 `property.data_type`）中提取数据类型。
2. **类型判断与选项分配**：
   - 如果是数字类型（包含 `int`, `integer`, `decimal`, `double`, `float`, `number`, `numeric` 等）：分配 `[等于, 不等于, 大于, 大于等于, 小于, 小于等于]`。
   - 如果是时间类型（包含 `datetime`, `timestamp`, `date`）：分配 `[等于, 不等于, 大于, 大于等于, 小于, 小于等于]`。
   - 其他（默认作为字符串类型）：分配 `[等于, 不等于, 包含, 不包含, 以...开头, 以...结尾]`。
3. **合法性校验**：当字段切换导致数据类型变化时，会调用 `ensureOperatorValid` 方法，检查当前选中的操作符是否在新的可用选项列表中，如果不在，则自动回退到列表的第一个操作符。

### 2.2 存在的问题与局限性
1. **逻辑耦合严重**：`getOperatorOptions` 和 `ensureOperatorValid` 这些纯粹的逻辑计算函数，被硬编码在具体的 UI 组件内部（如 `GlobalFilterComponent`），导致在事件分析、留存分析等其他地方无法复用，或存在大量重复代码。
   - **解决方案**：将操作符的推导和校验逻辑剥离出 UI 组件，封装为独立的 Composable 函数 (`useFilterOperator`)，实现逻辑的集中管理与多组件复用。
2. **类型判断不够收敛**：数字类型的别名判断过多（8种），实际上在向前端传递元数据时，后端或者 API 层就应该将这些数据库底层类型统一归一化为标准的 `DataType` 枚举（如 `number`）。
   - **解决方案**：在获取到后端元数据时（或在更底层的 API 封装处），统一进行类型映射，将各种底层数据库类型（如 `int`, `decimal`, `float` 等）收敛为前端标准的 `DataType.Number`。前端组件仅依赖标准的 `DataType` 枚举进行判断。
3. **缺失重要操作符**：
   - 数字和时间类型缺失了非常重要的 `BETWEEN`（区间）操作符。
   - 字符串类型缺失了 `为空` (IS NULL) 和 `不为空` (IS NOT NULL) 操作符。
   - 没有支持“相对时间”（如动态日期、相对事件发生时刻）的高级操作符规划。
   - **解决方案**：在新的操作符映射矩阵中，明确将 `BETWEEN` 加入数字和时间类型的支持列表；将 `IS NULL` 和 `IS NOT NULL` 加入字符串和数字类型的支持列表；并预留对高级时间操作符的扩展空间。

## 3. 重构目标与新设计

在新版 `dmp_admin_v2` 中，我们需要将操作符的推导逻辑抽离为一个纯函数或者 Composable，并与 `src/types/doris/common.ts` 中定义的 `Operator` 和 `DataType` 枚举深度绑定。

### 3.1 核心设计原则
- **数据驱动**：完全基于 `DataType` 枚举来推导可用的 `Operator` 列表。
- **高内聚低耦合**：将操作符配置和推导逻辑集中管理在 `constants/operators.ts` 或 `composables/useOperators.ts` 中。
- **自动纠偏机制**：提供标准化的 `validateOperator` 方法，在数据类型发生切换时，自动纠正非法操作符。

### 3.2 类型与操作符映射矩阵 (Operator Matrix)

我们需要构建一个基于 `DataType` 的操作符支持矩阵：

| 数据类型 (DataType) | 支持的操作符 (Operator) | 默认操作符 |
|---|---|---|
| **String** (`string`) | `等于`, `不等于`, `包含`, `不包含`, `为空`, `不为空`, `以...开头`, `以...结尾` | `等于` |
| **Number** (`number`) | `等于`, `不等于`, `大于`, `大于等于`, `小于`, `小于等于`, `区间(Between)`, `为空`, `不为空` | `等于` |
| **DateTime** (`datetime`) | `等于`, `不等于`, `大于`, `大于等于`, `小于`, `小于等于`, `区间(Between)`, `为空`, `不为空`, `动态日期(规划中)` | `区间(Between)` |
| **Boolean** (`boolean`) | `等于`, `不等于` | `等于` |

## 4. 接口契约与开发规划

### 4.1 定义与补充常量
在 `src/types/doris/common.ts` (或新增 `src/constants/operators.ts`) 中，建立映射关系：

```typescript
// 定义操作符的展示文本
export const OperatorLabels: Record<Operator, string> = {
  [Operator.EqualTo]: '等于',
  [Operator.NotEqualTo]: '不等于',
  [Operator.Contains]: '包含',
  // ... 其他标签
}

// 定义不同 DataType 支持的操作符列表
export const DataTypeOperators: Record<DataType, Operator[]> = {
  [DataType.String]: [Operator.EqualTo, Operator.NotEqualTo, Operator.In, Operator.NotIn, ...],
  [DataType.Number]: [Operator.EqualTo, Operator.NotEqualTo, Operator.GreaterThan, Operator.Between, ...],
  [DataType.DateTime]: [Operator.Between, Operator.GreaterThan, Operator.LessThan, ...],
  [DataType.Boolean]: [Operator.EqualTo, Operator.NotEqualTo]
}
```

### 4.2 抽离 Composable (`useFilterOperator.ts`)
创建一个通用的 Hook，专门处理操作符的动态加载与校验：

```typescript
export function useFilterOperator(dataTypeRef: Ref<DataType>) {
  // 计算当前数据类型可用的操作符选项
  const availableOperators = computed(() => {
    const ops = DataTypeOperators[dataTypeRef.value] || DataTypeOperators[DataType.String];
    return ops.map(op => ({ value: op, label: OperatorLabels[op] }));
  });

  // 校验并纠正当前操作符
  const validateOperator = (currentOperator: Operator): Operator => {
    const allowed = availableOperators.value.map(o => o.value);
    if (!allowed.includes(currentOperator)) {
      // 如果是非法操作符，返回该类型支持的第一个（或默认）操作符
      return allowed[0];
    }
    return currentOperator;
  };

  return {
    availableOperators,
    validateOperator
  };
}
```

### 4.3 实施步骤建议
1. **完善常量定义**：在前端补充完整的 `OperatorLabels` 和 `DataTypeOperators` 映射表。
2. **实现 `useFilterOperator`**：编写上述的 Composable 逻辑。
3. **改造现有的过滤器组件**（如 `MetricFilterEditor` 或全局过滤器）：
   - 移除组件内硬编码的 `getOperatorOptions`。
   - 引入 `useFilterOperator`。
   - 在字段 (`field`) 发生改变并带回新的 `dataType` 时，调用 `validateOperator` 修正当前的 `operator`。
   - 绑定 `<el-select>` 到 `availableOperators` 渲染下拉列表。
