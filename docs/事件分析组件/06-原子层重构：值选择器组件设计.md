# 06-原子层重构：值选择器组件设计

## 1. 概述
在事件分析、漏斗分析等模块中，过滤器（Filter）是核心组件之一。而**值选择器（Value Selector）**则是过滤器中最底层的原子组件，负责根据所选的“字段数据类型”与“操作符”，动态渲染最合适的输入控件（如下拉框、日期选择器、数字输入框等），并支持从后端动态加载字段的已有维度值供用户选择。

## 2. 现有实现分析（基于 `old_frontend`）

### 2.1 原逻辑回顾
在老版代码 `DimensionValueSelect.vue` 中，该组件主要实现了以下功能：
1. **时间类型处理**：
   - 依赖传入的 `dataType`，或者通过字段名（如包含 `time`, `date`）进行启发式判断。
   - 当操作符为 `BETWEEN` 时，渲染 `<el-date-picker type="datetimerange">`。
   - 其他操作符渲染 `<el-date-picker type="datetime">`。
2. **常规类型处理（字符串/数字）**：
   - 统一使用 `<el-select>` 组件，开启 `allow-create` 和 `filterable`（支持手填和搜索）。
   - 如果操作符为 `IN` 或 `NOT IN`（或显式传入 `multiple`），则开启多选模式。
   - 在 `focus` 时触发 `dimensionsApi.listValues` 接口，去服务端拉取当前字段的去重维度值。
3. **值形态转换**：
   - 通过 `watch` 监听 `operator` 的变化，在单选（字符串）与多选（数组）之间来回转换 `innerValue`。

### 2.2 存在的问题与局限性
1. **类型判断不够严谨**：严重依赖字段名的字符串匹配（如 `.endsWith('time')`），容易导致误判。
2. **缺乏对数字和布尔类型的精细化支持**：
   - 数字类型没有使用 `<el-input-number>`，可能导致用户输入非数字字符。
   - 布尔类型没有提供明确的 True/False 选项，依赖后端的维度值枚举。
3. **状态同步复杂**：内部维护了 `innerValue`, `dateSingleValue`, `dateRangeValue` 多个状态，且频繁相互同步，代码难以维护。
4. **API 强耦合**：数据获取逻辑直接写在组件内部，复用性差，且没有良好的缓存机制。

## 3. 重构目标与新设计

在新版 `dmp_admin_v2` 中，我们将该组件命名为 `FilterValueInput.vue`（或 `DimensionValueSelect.vue` 的升级版），遵循 Vue 3 Composition API 规范进行重构。

### 3.1 核心设计原则
- **严格的数据类型驱动**：完全依赖传入的 `dataType`（`string`, `number`, `datetime`, `boolean`）来决定渲染控件，彻底废弃按字段名猜测的逻辑。
- **操作符驱动的值形态**：严格根据当前操作符的基数（如 `==` 是标量，`in` 是数组，`between` 是长度为 2 的数组）来约束和转换 `modelValue`。
- **逻辑抽离**：将“动态获取维度值”的逻辑抽离为独立的 Composable（如 `useDimensionValues`），提升可测试性与复用性。

### 3.2 渲染策略矩阵

| 数据类型 (DataType) | 操作符 (Operator) | 渲染组件 (Element Plus) | 值类型 | 备注 |
|---|---|---|---|---|
| **string** | `=, !=, contains...` | `<el-select allow-create filterable>` | `string` | 单选，支持远程搜索维度值 |
| **string** | `in, not_in` | `<el-select multiple allow-create>` | `string[]` | 多选，支持远程搜索维度值 |
| **number** | `=, !=, >, <, >=, <=` | `<el-input-number>` | `number` | 标量数字输入 |
| **number** | `in, not_in` | `<el-select multiple allow-create>` | `number[]` | 数字标签输入，需做数字校验 |
| **number** | `between` | 两个 `<el-input-number>` + `-` 连字符 | `[number, number]` | 区间输入 |
| **datetime** | `=, !=, >, <, >=, <=` | `<el-date-picker type="datetime">` | `string` | 单个时间点选择 |
| **datetime** | `between` | `<el-date-picker type="datetimerange">` | `[string, string]` | 时间范围选择 |
| **boolean** | `=, !=` | `<el-select>` (固定选项 True/False) | `boolean` | 不允许自定义输入 |

## 4. 接口契约 (Interface)

### 4.1 Props 定义
```typescript
export interface FilterValueInputProps {
  modelValue: string | number | boolean | any[] | null;
  dataType: 'string' | 'number' | 'datetime' | 'boolean';
  operator: string; // 如 'eq', 'in', 'between', 'gt' 等枚举值
  
  // 用于动态加载维度值的上下文
  projectAlias?: string;
  tableName?: string; // 'event_data' | 'user_data'
  fieldName?: string;
  eventId?: string;   // 仅事件属性需要
  
  placeholder?: string;
  disabled?: boolean;
}
```

### 4.2 Emits 定义
```typescript
export interface FilterValueInputEmits {
  (e: 'update:modelValue', value: any): void;
}
```

## 5. 组件应用场景

为了保证全站过滤体验的一致性，`FilterValueInput.vue` 组件需要被设计为高度通用，并在以下核心场景中无缝应用：

1. **事件过滤器 (Event Filter)**：
   - 场景：在事件分析、漏斗分析、留存分析中，针对单一事件（如 `sys.login`）设置的局部条件。
   - 特点：需要传入 `eventId` 供后端精准拉取该事件下的属性维度值。

2. **全局过滤器 (Global Filter)**：
   - 场景：在各个分析模型的顶部（GlobalFiltersCard），作用于当前分析模型下所有事件的通用条件。
   - 特点：通常基于 `user_data` 或所有事件的公共属性，不需要传入特定的 `eventId`。

3. **看板全局过滤器 (Dashboard Global Filter)**：
   - 场景：在数据看板（Dashboard）顶部，用户可动态调节的筛选条件，作用于看板内的所有图表卡片。
   - 特点：上下文可能较为复杂，需要组件具备高度的容错性，并在切换不同项目的看板时能够及时清除缓存和重置状态。

## 6. 后端 API 接口契约

为了支持动态拉取字段维度值，后端需提供以下接口：

**接口路径**：`POST /api/v1/analytics/dimension` (基于 `dmp_admin_v2/backend/internal/api/router.go`)

**Request Payload (`DimensionReq`)**:
```json
{
  "project_alias": "test_project",
  "table": "event_data", // 或 "user_data" / "user_tag_data"
  "field": "e_os",       // 需要获取维度值的列名
  "e_event_id": "sys.login" // 可选，如果存在则只拉取该事件下的维度值
}
```

**Response**:
```json
{
  "code": 0,
  "data": [
    { "value": "iOS" },
    { "value": "Android" }
  ]
}
```
*注：后端返回的键名为 `value`（通过 `SELECT DISTINCT field AS value` 实现），前端 `useDimensionValues` 中需兼容取值。*

## 7. 开发步骤规划

1. **基础类型与工具准备**：
   - 确保 `src/types/doris/filter.ts` 中拥有完整的 `Operator` 和 `DataType` 枚举。
   - 实现一个纯函数 `normalizeFilterValue(value, operator, dataType)`，用于在操作符切换时安全地转换值（例如从 `['A', 'B']` 切换为 `=` 时，自动取 `A`）。
2. **抽离数据拉取逻辑 (`useDimensionValues.ts`)**：
   - 封装对 `/v1/analytics/dimension` 接口的调用。
   - 增加防抖（Debounce）和简单的内存缓存（以 `project_alias+table+field` 为 Key），避免频繁聚焦导致重复请求。
3. **编写 `FilterValueInput.vue` 组件模板**：
   - 使用 `v-if/v-else-if` 结合 `dataType` 和 `operator` 分支渲染不同的 Element Plus 组件。
4. **状态同步与事件处理**：
   - 监听 `operator` 的变化，调用 `normalizeFilterValue` 修正 `modelValue` 并向外 `emit`。
   - **触发维度值加载的核心事件**：
     - 处理 `<el-select>` 的 `@focus` 事件，确保输入框获得焦点时发起维度值加载请求。
     - 处理 `<el-select>` 的 `@visible-change` 事件，确保用户点击展开下拉面板时，如果尚未加载数据也能触发加载。
     - *注意：不要只局限于 `DataType.String`，当数字类型(`DataType.Number`) 作为多选（`IN`/`NOT IN`）时，同样需要渲染成支持下拉的 `<el-select>`，并触发数据拉取。*
5. **单元测试与集成联调**：
   - 确保在父组件（如 `MetricFilterEditor` 或 `GlobalFiltersCard`）中切换操作符时，输入框能平滑过渡且不会抛出类型警告。
