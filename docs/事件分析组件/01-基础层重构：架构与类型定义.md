# 01-基础层重构：架构与类型定义

本计划聚焦于事件分析模块的基础类型定义与全局常量规范化，为后续重构提供坚实的类型安全保障。

---

## 1. 目标
- 消除 `any` 类型，建立完整的分析业务接口体系。
- 规范化业务常量，减少硬编码。
- 确保整个重构过程中的代码可读性与健壮性。

## 2. 执行步骤

### 步骤 1: 完善分析业务类型定义
在 `src/types/doris/analysis.ts` (或对应路径) 中定义以下接口：
- `AnalysisMetric`: 定义基础指标结构（事件ID、字段、公式、别名、过滤条件等）。
- `AnalysisFilter`: 定义过滤条件结构（表名、字段名、操作符、值、类型等）。
- `AnalysisGroup`: 定义分组维度结构（类型、列信息、区间配置等）。
- `AnalysisPayload`: 统一定义发送给后端的请求体结构。

### 步骤 2: 提取全局业务常量
在 `src/constants/analysis.ts` 中提取以下硬编码字符串：
- `TableNames`: `EVENT_TABLE`, `USER_TABLE`, `TAG_TABLE` 等。
- `FormulaTypes`: 指标计算公式的枚举。
- `OperatorTypes`: 过滤操作符的枚举。
- `VizModes`: `line`, `bar`, `pie`, `table` 等可视化模式。

### 步骤 3: 全局类型替换
- 在 `useEventAnalysisForm.ts` 中，将 `form` 对象的类型从 `any` 或部分定义改为 `AnalysisPayload`。
- 在组件 `Props` 中开始引用新定义的接口。

## 3. 验证标准
- [ ] 编译通过，无类型冲突。
- [ ] 代码中不再出现未定义的硬编码业务字符串（如特定的表名）。
- [ ] IDE 能够对 `form` 对象及其子属性提供精准的自动补全。
