# 漏斗分析 - 数据结构与接口契约 (old_frontend)

本文档整理了 `old_frontend` 中漏斗分析模块的数据结构及其 API 交互细节。

---

## 1. 核心数据结构

### 1.1 `FunnelForm` (前端表单结构)
```typescript
interface FunnelForm {
  project_alias: string;
  time_zone: string;
  
  // 漏斗步骤 (2-8 步)
  event_metrics: FunnelStepMetric[];

  // 时间粒度 + 转化窗口
  time_grain: {
    column: { table: string; field: string; alias: string };
    interval: number; // 1-Hour, 2-Day, 3-Week, 4-Month
    window_num: number; // 转化窗口期（天）
  };

  // 全局过滤
  global_filter_groups: {
    global_filters: {
      scope: number; // 1-And, 0-Or
      filters: any[];
      tag_filters: any[];
      user_group_filters: any[];
    };
    query_dates?: string[]; // [startDate, endDate]
  };

  // 分组维度
  groups: any[];

  page_size: number;
  page_num: number;
}
```

### 1.2 `FunnelStepMetric` (单个步骤结构)
```typescript
interface FunnelStepMetric {
  _uid: number; // 前端唯一标识
  type: number; // 1-Normal
  e_event_id: string; // 事件 ID
  name: string; // 步骤显示名称
  metric: {
    column: { table: string; field: string };
    formula: number; // MF.BITMAP_UNION (18)
    format: string; // 'int'
  };
  filter_group: {
    scope: number;
    filters: any[];
  };
}
```

## 2. API 接口契约

### 2.1 执行漏斗分析
-   **URL**: `/api/v1/funnel-analysis/query` (示例)
-   **Method**: `POST`
-   **Request Payload**: 同 `FunnelForm`。
-   **Response Structure**:
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "rows": [
      {
        "group_values": ["Value1", "Value2"], // 分组值（如有）
        "step_0_count": 1000, // 第 1 步人数
        "step_1_count": 500,  // 第 2 步人数
        "step_1_rate": 0.5,   // 第 2 步转化率
        "step_2_count": 250,  // 第 3 步人数
        "step_2_rate": 0.5    // 第 3 步转化率
      }
    ],
    "columns": ["分组1", "分组2", "步骤1", "步骤2", "步骤2转化率", "步骤3", "步骤3转化率"],
    "total": 1,
    "sql": "SELECT ..."
  }
}
```

## 3. 关键字段映射

-   **Formula (MF.BITMAP_UNION)**：漏斗分析中固定使用 18（Bitmap Union），后端利用此公式进行精确的去重计数。
-   **Window Num**：核心业务参数，定义了完成整个漏斗流程的最大允许天数。
-   **Step Proxy**：前端在编辑过滤器时，会使用 `getStepProxy(index)` 来获取步骤对象，确保 `MetricFilterEditor` 能正确绑定到具体的步骤过滤配置上。
-   **UnwrapEnvelope**：由于后端响应可能经过多层 `data` 包装，前端使用通用的解包函数来确保无论后端结构如何变化，都能正确提取核心数据。
