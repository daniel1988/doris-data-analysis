# 05-数据看板全链路流程

## 1. 流程概述
数据看板的功能逻辑涵盖了看板的创建、报表的关联、布局的动态调整以及数据的聚合展示。

## 2. 核心业务流程

### 2.1 创建看板流程
1. **用户操作**：点击“看板”页面侧边栏的“新建”按钮。
2. **前端交互**：弹出对话框，输入看板名称、显示名称和描述。
3. **API 调用**：发送 `POST /api/v1/dashboard` 请求。
4. **后端处理**：
   - 校验项目别名是否有效。
   - 在 `dashboards` 表中插入新记录。
   - 返回新创建的看板对象（含 ID）。
5. **前端响应**：刷新左侧看板列表，并自动选中新创建的看板。

### 2.2 将报表添加至看板流程 (分析视图触发)
1. **用户操作**：在“事件分析/漏斗分析”等页面点击“保存”。
2. **前端交互**：在保存对话框中，除了填写报表信息，还可选择一个已有的看板。
3. **保存报表**：调用 `POST /api/v1/report` 保存报表。
4. **关联看板**：
   - 若用户选择了看板，前端在报表保存成功后获取 `report_id`。
   - 调用 `POST /api/v1/dashboard/item`。
   - **默认布局逻辑**：
     - 查询目标看板已有的 Items。
     - 计算新 Item 的 `position_y`（通常放在当前最大 Y 值下方）。
     - 设置默认宽度（如 12 或 24）。
5. **后端处理**：在 `dashboard_items` 表中插入关联记录。

### 2.3 看板展示与数据加载流程
1. **路由跳转**：用户点击侧边栏看板。
2. **加载详情**：调用 `GET /api/v1/dashboard/:id`（后端使用 Preload 加载 Items）。
3. **渲染网格**：前端 `GridLayout` 根据 `DashboardItem` 的坐标渲染占位卡片。
4. **并发加载数据**：
   - 每个 `DashboardItemCard` 独立请求对应的报表定义（`GET /api/v1/report/:id`）。
   - 获取报表定义后，解析 `query_params`。
   - 合并看板全局筛选器（可选）后，调用相应的分析 API（如 `/api/v1/analytics/event`）。
   - 渲染 `ChartView`。

### 2.4 布局调整与保存流程
1. **进入编辑模式**：用户点击“编辑布局”。
2. **拖拽/缩放**：用户调整组件位置和大小，前端 `layout` 响应式更新。
3. **保存布局**：
   - 用户点击“保存布局”。
   - 前端遍历 `layout` 数组，循环调用 `PUT /api/v1/dashboard/item` 更新坐标。
   - 退出编辑模式。

### 2.5 看板内直接添加报表流程 (新增功能)
1. **用户操作**：在“看板”详情页点击“添加报表”按钮。
2. **前端交互**：弹出报表搜索选择框，展示当前项目下所有已保存的报表。
3. **选择报表**：用户选择一个或多个报表。
4. **批量添加**：前端调用 `POST /api/v1/dashboard/item`（或建议的批量接口）。
5. **布局计算**：自动将新报表排列在看板末尾。

## 3. 常见异常与排查思路
- **新建看板无法添加报表**：
  - **原因 1：ID 丢失**。后端创建报表后未正确返回自增 ID，导致前端 `savedReport.id` 为空。
  - **原因 2：异步同步延迟**。看板创建后，Doris 元数据同步可能存在极短延迟，立即添加 Item 可能触发短暂的“看板不存在”错误。
  - **原因 3：前端参数绑定错误**。`SaveReportDialog` 中的 `dashboardId` 未能正确传递给 `saveReport` 函数。
- **布局保存失败**：
  - **原因：LIMIT 语法错误**。GORM 在执行单个记录更新时自动追加 `LIMIT 1`，需在 Service 层使用切片模式规避。
