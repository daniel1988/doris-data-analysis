# 06-报表展示与数据加载逻辑详解

## 1. 报表数据执行链路
看板中报表的展示不仅仅是简单的接口调用，涉及多层参数合并与标准化处理。

### 1.1 参数标准化 (Normalization)
在调用分析接口前，需对报表保存的 `query_params` 进行标准化处理，以兼容 Doris 后端逻辑：
- **指标映射**：
  - `__TOTAL_TIMES__` (总次数) -> 映射为 `e_openid` 字段，配合 `Formula: 2` (COUNT)。
  - `__TOTAL_USERS__` (总人数) -> 映射为 `e_openid` 字段，配合 `Formula: 4` (APPROX_COUNT_DISTINCT)。
- **自定义指标处理**：
  - 若指标类型为 `2` (自定义指标)，需递归处理其内部的 `event_metrics`。
  - 自动生成内部变量名（如 `metric_0`, `metric_1`）用于公式计算。
- **时间粒度规范化**：
  - 确保包含 `table` (默认 `event_data`)、`field` (默认 `e_event_time`) 及 `interval` (默认 `2` - 按天)。

### 1.2 过滤器三层合并逻辑
看板报表的最终过滤条件由以下三部分合并而成：
1.  **报表级过滤 (Report Filters)**：保存报表时自带的固定过滤条件。
2.  **看板级全局过滤 (Dashboard Saved Filters)**：看板配置中保存的默认筛选器（如：默认只看某个渠道的数据）。
3.  **实时 UI 过滤 (Current UI Filters)**：用户在查看看板时，通过顶部筛选栏临时选择的条件。

**合并算法**：
- `global_filters` = `Report.global_filters` + `Dashboard.saved_filters`。
- `dashboard_form_filters` = `UI.current_filters`。
- 后端会将这两部分通过 `AND` 逻辑进行最终聚合。

### 1.3 时间范围处理
- **常规图表**：使用看板顶部选择的全局时间范围。
- **指标卡 (Small Charts)**：
  - 为了计算 **环比 (Period-over-Period)**，前端需自动推算“上一周期”的时间。
  - 例如：当前选择 3月10日-3月17日（8天），则上一周期为 3月2日-3月9日。
  - 请求数据时，时间范围会自动扩大以覆盖这两个周期。

## 2. 布局展示逻辑

### 2.1 组件尺寸分类
系统支持三种标准尺寸，决定了在栅格系统中的占用比例：
- **小图 (Small/Stat)**：通常占用 3/12 或 6/24 宽度。用于展示核心 KPI、环比增长等汇总数据。
- **中图 (Medium)**：通常占用 6/12 或 12/24 宽度。适用于折线图、柱状图。
- **大图 (Large)**：占用 12/12 或 24/24 全宽。适用于明细表格、漏斗图或复杂的留存矩阵。

### 2.2 报表展示模式与切换 (Display Modes & Switching)
为了统一看板展示风格并提升数据可读性，对各分析模型的报表展示逻辑进行如下优化：

- **默认展示逻辑 (Default Display)**：
  - **通用模型**：包括事件分析、漏斗分析、属性分析、分布分析。进入看板时，**默认以“数据表 (Data Table)”形式展示**。
  - **留存分析**：由于留存矩阵的特殊性，**仅支持“数据表 (Matrix)”展示**，不提供图表切换功能。
- **图表切换能力 (Chart Toggling)**：
  - 对于通用模型（事件、漏斗、属性、分布），卡片右上角操作栏提供“图表/表格”切换按钮。
  - 用户点击切换后，根据报表保存时的 `vizMode` 配置（如折线、柱状、饼图等）渲染对应的图表组件。
- **展示组件要求**：
  - **表格模式**：复用分析引擎返回的 `rows` 与 `columns` 数据，支持前端排序、分页及列宽自适应。
  - **图表模式**：复用分析页面的 `ChartView` 组件，确保看板内图表样式与分析页一致。
- **汇总指标卡 (KPI Card)**：保持原有的大数字展示逻辑，不参与表格/图表切换。
- **状态保持**：当前的切换状态（表格或图表）仅在前端运行时生效，页面刷新后将重新恢复为默认的“数据表”展示。

### 2.3 响应式策略
- 在 `grid` 布局模式下，根据屏幕宽度动态调整 `col-num`。
- 小屏设备下，所有组件强制设为 24 宽（全宽铺满）。

## 3. 汇总指标卡 (KPI Card) 特殊处理
对于 category 为 `stat` 或尺寸为 `small` 的报表，执行以下逻辑：
1.  **数据截取**：从返回的趋势数据中，取最后一行作为“当前值”。
2.  **环比计算**：
    - `change = current_value - previous_value`
    - `percentage = (change / previous_value) * 100%`
3.  **可视化渲染**：展示大数字标题，下方配以带颜色的箭头（红涨绿跌或绿涨红跌，视指标含义而定）。

## 4. 事件分析报表在看板中“无数据”的专项说明

### 4.1 现象定义
- 看板卡片出现“暂无数据”或“数据加载解析失败”。
- 同一个报表在分析页面可正常计算，但进入看板后不展示结果。

### 4.2 必要前提与约束
为保证事件分析报表可在看板中加载，需满足以下约束：
- `report.category` 必须为 `events`（与看板数据分发逻辑一致）。
- `report.query_params` 必须包含 `form.event_metrics`、`form.filter_groups`、`form.time_grain`。
- `project_alias` 必须有效，且该项目下元数据（事件/属性）可查询。

### 4.3 典型失败点
- **分类不匹配**：保存报表时 `category` 不是 `events`，看板无法命中事件分析执行分支。
- **参数结构缺失**：`query_params` 缺字段或字段类型异常，导致 payload 构建失败。
- **全局筛选冲突**：看板筛选器与报表原始筛选叠加后，无数据命中。
- **前端模块导出不完整**：属性或事件选择器 API 方法缺失，触发运行时异常并中断渲染。

### 4.4 建议排查顺序
1. 先确认 `report.category` 是否为 `events`。
2. 再检查 `query_params` 是否包含完整 `form` 结构。
3. 检查看板当前全局筛选条件是否导致数据被过滤为空。
4. 打开浏览器 Console，排查模块导出、JSON 解析、payload 构建相关错误。

## 5. 报表名跳转分析模型与配置回显

### 5.1 目标行为
- 在看板点击“编辑报表”或点击报表名称后，跳转至对应分析模型页面。
- 跳转后自动加载该报表配置，并回显指标、过滤、时间范围、分组、图表形态。

### 5.2 跳转参数约定
- 路由参数：`reportId`。
- 推荐示例：`/analytics/event?reportId=123`。

### 5.3 回显执行链路（设计约定）
1. 分析页面 `onMounted` 读取 `route.query.reportId`。
2. 调用报表详情接口获取 `query_params`。
3. 使用页面表单加载函数回填配置（而非逐字段手动赋值）。
4. 回填完成后自动触发一次分析请求，保证“跳转即见结果”。

### 5.4 当前文档补充结论
- 本文档已补充“看板无数据”与“跳转回显”链路说明。
- 该部分仍属于需落地实现和联调的功能要求，需在前端与后端对齐后按本约定验收。

## 6. 验收口径（文档先行）
- **看板展示**：事件分析报表进入看板后可稳定展示数据，不出现空白卡片。
- **跳转能力**：可从看板项定位到对应分析模型页。
- **配置回显**：跳转后指标、过滤、时间范围与保存时报表配置一致。
- **结果一致**：看板卡片与分析页在相同筛选条件下结果一致。

## 7. old_frontend 对照链路（ViewDashboard）

### 7.1 页面主流程
基于 [ViewDashboard.vue](file:///d:/gitee/dmp_admin_v2/old_frontend/views/dashboard/ViewDashboard.vue#L1-L93) 与 [useDashboardView.ts](file:///d:/gitee/dmp_admin_v2/old_frontend/views/dashboard/view/composables/useDashboardView.ts#L13-L329)，完整流程为：
1. 路由进入看板详情页，读取 `route.params.id`。
2. 请求看板详情并回填看板元信息、布局、组件列表、过滤器配置。
3. 解析两类过滤器：
   - `global_filters`（看板保存的全局过滤）
   - `dashboard_form_filters`（用户在看板页实时配置的过滤）
4. 逐个组件串行加载报表数据（包含轻量间隔，避免并发压垮接口）。
5. 根据 `refresh_interval` 与开关状态启动自动刷新。
6. 渲染 Header / Controls / Content / FilterDialog 四层结构。

### 7.2 组件级数据执行链路
基于 [data-execution.ts](file:///d:/gitee/dmp_admin_v2/old_frontend/views/dashboard/view/composables/internal/data-execution.ts#L43-L342)：
1. 通过 `reportId` 获取报表定义与 `query_params`。
2. 按 `category` 选择执行器：
   - `events` -> `eventAnalysisV1Api.execute`
   - `funnel` -> `funnelAnalysisApi.query`
   - `retention` -> `retentionAnalysisApi.query`
   - `scatter` -> `scatterAnalysisApi.execute`
   - `property` -> `userPropertyAnalysisApi.execute`
   - `sql` -> `sqlApi.executeQuery`
3. 构建请求时统一注入：
   - 时间范围（支持组件级覆盖）
   - 全局过滤与实时过滤合并
   - 分页参数（`page_num/page_size`）
4. 解析 Doris 响应，标准化为 `rows + columns + count`。
5. 小图（small）对事件分析结果追加“当前值+环比”计算。

### 7.3 报表标题跳转与配置回显入口
基于 [ComponentCard.vue](file:///d:/gitee/dmp_admin_v2/old_frontend/components/dashboard/ComponentCard.vue#L20-L351)：
- 点击报表标题会按 `reportType` 跳转分析页，并携带：
  - `reportId`
  - `project_alias`
- 小图卡片点击同样支持跳转。
- 该链路是“看板 -> 分析模型回显”的关键入口，文档与实现需保持一致。

### 7.4 稳定性与运行期保护
基于 [useDashboardView.ts](file:///d:/gitee/dmp_admin_v2/old_frontend/views/dashboard/view/composables/useDashboardView.ts#L277-L310)：
- 监听网络异常事件，自动暂停刷新并展示横幅提醒（30 秒后恢复）。
- 监听项目切换，若当前看板项目与全局项目不一致，终止加载并重定向。
- 组件支持局部时间切换、局部分页、局部刷新，不依赖整页重载。
