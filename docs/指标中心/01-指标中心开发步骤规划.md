# 指标中心开发规划与设计文档

## 1. 背景与目标

当前系统中，事件分析、AI 智能分析等模块均直接基于底层明细表 (`event_data`、`user_data`) 进行查询和分析。随着业务复杂度的增加，衍生出了大量常用的业务指标（例如 DAU、ARPU、留存率、转化率等）。为了避免：
1. 分析人员反复编写复杂的 SQL 或配置；
2. AI 智能分析时对业务口径理解偏差（如 DAU 是哪些事件的去重）；
3. 口径不一致导致的数据孤岛。

我们需要搭建一个**指标中心 (Metrics Center)**，允许管理员或数据分析师通过配置界面的形式，将复杂的业务口径（公式、过滤条件）封装为标准化的**指标 (Metric)**。

**核心目标：**
- **统一口径**：在界面配置衍生指标（例如 `DAU` 的公式为一条完整的 SQL 聚合片段：`COUNT(DISTINCT CASE WHEN e_event_id='sys.login' THEN e_openid END)`）。
- **赋能 AI**：将指标中心定义的业务指标暴露给 AI（即注入到 Virtual Schema 中），使得大模型在生成多指标报表或回答自然语言问题时，直接引用这些高阶指标，降低 SQL 生成难度和错误率。
- **可复用**：指标一旦配置，可在看版 (Dashboard)、事件分析、AI 分析等多个模块无缝复用。

## 2. 核心概念与数据模型设计

### 2.1 核心概念定义

- **基础字段 (Field)**：`event_data` 和 `user_data` 中的原生字段。
- **指标 (Metric)**：由基础字段、聚合函数和条件表达式计算出的具有业务含义的数值。
  - **名称**：DAU (日活跃用户数)
  - **公式/表达式**：`COUNT(DISTINCT CASE WHEN e_event_id='sys.login' THEN e_openid END)`
  - **业务类型**：用户活跃、商业化、留存等

### 2.2 数据库表设计规划

需要在 `dmp_center` 数据库中新增一张指标配置表 `project_metrics`：

```sql
CREATE TABLE IF NOT EXISTS dmp_center.project_metrics (
  `project_alias`   VARCHAR(58) NOT NULL COMMENT '项目别名',
  `metric_code`     VARCHAR(128) NOT NULL COMMENT '指标英文标识 (如: dau_count)',
  `metric_name`     VARCHAR(128) NOT NULL COMMENT '指标名称 (如: DAU)',
  `expression`      TEXT NOT NULL COMMENT '指标计算表达式 (如: COUNT(DISTINCT CASE WHEN e_event_id=''sys.login'' THEN e_openid END))',
  `base_table`      VARCHAR(64) NOT NULL DEFAULT 'event_data' COMMENT '基于哪张表 (event_data/user_data)',
  `description`     TEXT NULL COMMENT '业务口径说明',
  `status`          TINYINT NOT NULL DEFAULT 1 COMMENT '状态: 1启用 0停用',
  `create_time`     DATETIME DEFAULT CURRENT_TIMESTAMP,
  `update_time`     DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=OLAP
UNIQUE KEY(`project_alias`, `metric_code`)
DISTRIBUTED BY HASH(`project_alias`, `metric_code`) BUCKETS 1
PROPERTIES (
  "enable_unique_key_merge_on_write" = "true"
);
```

## 3. 开发步骤分解

### 阶段一：后端指标管理 API 及数据库支持
**目标**：完成对指标的增删改查 (CRUD) 接口，并能在中心库管理配置。

1. **数据库初始化更新**：
   - 在 `docs/sql/dmp_center.sql` 和 `db_init/main.go` 相关的初始化脚本中追加 `project_metrics` 的建表语句。
2. **生成 GORM 模型与 Service 层**：
   - 创建 `internal/model/project_metric.go`。
   - 创建 `internal/service/metadata/metric_service.go` 提供增删改查逻辑。
3. **暴露 RESTful API**：
   - 增加 `GET /api/metadata/metrics` (列表查询)。
   - 增加 `POST /api/metadata/metrics` (创建指标)。
   - 增加 `PUT /api/metadata/metrics/:id` (修改指标)。
   - 增加 `DELETE /api/metadata/metrics/:id` (删除指标)。

### 阶段二：前端「指标中心」管理界面开发
**目标**：为数据分析师提供一个直观的配置界面。

1. **路由与菜单配置**：
   - 在 `src/router/index.ts` 的 `metadata` 模块下新增 `/metadata/metrics` 路由。
   - 在 `SidebarLayout.vue` 的「元数据」菜单下追加「指标中心」入口。
2. **列表页与配置表单 (`views/metadata/Metrics.vue`)**：
   - **列表页**：展示当前项目下配置的所有指标，包含名称、代码、状态。
   - **新增/编辑弹窗**：
     - 提供基础信息的输入（名称、标识）。
     - 提供**公式编辑器**（输入完整的聚合SQL片段，如：`COUNT(DISTINCT case when ...)`）。
3. **API 联调**：接入阶段一完成的增删改查接口。

### 阶段三：将指标注入 AI 智能分析 (Chat2SQL 赋能)
**目标**：让 AI 知道如何使用这些已定义的业务指标。

1. **改造 `ai_chat_service.go -> BuildVirtualSchema`**：
   - 在构建提供给 LLM 的 Schema prompt 时，除了查询 `project_event` 和 `project_property` 外，额外查询当前启用的 `project_metrics`。
   - 将指标信息格式化拼接进 Prompt 中。
   - **Prompt 拼接示例**：
     ```text
     Available Business Metrics (You can use these definitions directly in your SQL):
     - Metric: DAU (dau_count)
       Logic: COUNT(DISTINCT CASE WHEN e_event_id='sys.login' THEN e_openid END)
       Description: 日活跃用户数
     - Metric: Total Revenue (total_revenue)
       Logic: SUM(CASE WHEN e_event_id='sys.pay' THEN cast(e_properties->>'amount' as decimal) END)
       Description: 总充值金额
     ```
2. **优化 Prompt 引导规则**：
   - 告诉 LLM：当用户询问相关业务指标时（如“查询近七天的DAU”），请优先使用上述 `Available Business Metrics` 中提供的逻辑进行组合和查询。

### 阶段四：验证与扩展 (后续规划)
- **多指标报表生成测试**：通过自然语言提问“给我看最近 7 天的 DAU 和 充值总额趋势”，验证 AI 是否能正确套用指标中心里配置的 `case when` 公式。
- **与事件分析模块打通**（可选）：在传统的 UI 事件分析模块中，允许用户在“选择指标”的下拉框里直接选中“指标中心”配置好的衍生指标，后端自动将其解析为对应的 SQL 片段进行查询。

## 4. 预期使用工作流

1. **定义口径**：数据分析师在【元数据】-【指标中心】中新建指标，如：
   - 名称：`DAU`
   - 标识：`dau_count`
   - 表达式：`COUNT(DISTINCT case when e_event_id='sys.login' then e_openid end)`
2. **智能问答**：运营人员在【智能分析】输入：“按平台对比昨天的 DAU”
3. **AI 理解与生成**：AI 读取到 Schema 中的 DAU 定义，自动生成如下 SQL：
   ```sql
   SELECT 
     e_platform, 
     COUNT(DISTINCT case when e_event_id='sys.login' then e_openid end) as dau_count 
   FROM event_data 
   WHERE e_event_time >= '2023-10-01' AND e_event_time < '2023-10-02' 
   GROUP BY e_platform
   ```
4. **报表渲染**：前端拿到数据并渲染出对应图表。
