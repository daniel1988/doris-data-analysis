# 用户标签 - 数据结构与接口契约

本文档整理  用户标签模块中的核心数据结构、接口封装以及保存与构建阶段的请求契约，方便后续迁移时统一前后端协议。

## 1. 核心实体

用户标签主实体定义在：

- `old_frontend/api/userTag.ts`

### 1.1 `UserTag`

```ts
export interface UserTag {
  id: number;
  projectAlias: string;
  tagCode: string;
  tagShowName: string;
  tagType: TagType;
  description?: string;
  tagRule?: any;
  updateType: UpdateType;
  status: number;
  userCount?: number;
  createdBy: number;
  createdAt: string;
  updatedAt: string;
  creator?: {
    id: number;
    username: string;
    name: string;
  };
}
```

字段含义如下：

- `projectAlias`
  - 所属项目别名
- `tagCode`
  - 标签编码，唯一标识
- `tagShowName`
  - 标签展示名称
- `tagType`
  - 标签类型
- `tagRule`
  - 标签规则主体，不同类型结构不同
- `updateType`
  - 更新方式，静态或动态
- `status`
  - 标签是否启用
- `userCount`
  - 最新一次构建后的命中用户数

## 2. 标签类型与更新方式

### 2.1 `TagType`

```ts
export enum TagType {
  CONDITION = 'condition',
  FIRST_LAST = 'first_last',
  METRIC_VALUE = 'metric_value',
  ID = 'id',
  SQL = 'sql'
}
```

### 2.2 `UpdateType`

```ts
export enum UpdateType {
  STATIC = 'static',
  DYNAMIC = 'dynamic'
}
```

在页面语义上：

- `static`
  - 一般表示标签构建后结果固定
- `dynamic`
  - 表示标签会按规则持续更新

旧前端中多个规则编辑器还维护了更细的“自动更新时间”配置，但这些配置并没有在 `CreateUserTagRequest` 顶层显式展开，而是更多留存在各自的 `tagRule` 内部。

### 3.3 返回值兼容逻辑

`getUserTags()` 对旧接口做了兼容：

- `items` 不存在时，回退读取 `list`
- `pageSize` 不存在时，回退读取 `limit`

说明后端返回格式在历史上存在过变动，因此新实现如果要兼容旧服务，不能假设列表接口只有单一响应结构。

## 4. 创建与更新契约

### 4.1 创建请求

```ts
export interface CreateUserTagRequest {
  projectAlias: string;
  tagCode: string;
  tagShowName: string;
  tagType: TagType;
  description?: string;
  tagRule?: any;
  updateType?: UpdateType;
}
```

### 4.2 更新请求

```ts
export interface UpdateUserTagRequest {
  tagShowName?: string;
  description?: string;
  tagRule?: any;
  updateType?: UpdateType;
  status?: number;

```

创建和更新都围绕 `tagRule` 展开，这意味着：

- 标签规则协议是本模块最重要的可变部分
- 页面层必须按类型正确序列化规则

## 5. 各类型 `tagRule` 结构概览

### 5.1 条件标签

条件标签保存时，常见结构为数组，每个元素是一条标签值规则：

```ts
[
  {
    tag_value: '高活跃',
    set_operate: 1,
    event_filter_group: {
      set_operate: 1,
      event_filters: [...]
    },
    user_filter_group: {
      scope: 1,
      filters: [...]
    }
  }
]
```

关键点：

- 一个标签可有多个 `tag_value`
- 每个标签值分别对应独立条件
- `set_operate` 表示条件组组合关系

### 5.2 首末次标签

首末次标签在保存时会被序列化为 snake_case：

```ts
{
  first_or_last_tag_rule: {
    event_metric: {
      e_event_id: 'login',
      type: 1,
      metric: {
        column: { table: 'event_data', field: 'device_model' },
        formula: 18
      },
      filter_group: {
        scope: 1,
        filters: [...]
      }
    }
  },
  autoUpdateConfig: {...},
  timeFilter: {...}
}
```

其中：

- `formula = 18` 表示首次
- `formula = 17` 表示末次

### 5.3 指标值标签

指标值标签围绕 `event_metric` 构建：

```ts
{
  event_metric: {
    e_event_id: 'purchase',
    name: 'purchase',
    type: 1,
    metric: {
      column: { table: 'event_data', field: 'amount', alias: '' },
      formula: 1,
      format: 'raw'
    },
    filter_group: {
      scope: 1,
      filters: []
    }
  },
  global_filter_group: {
    scope: 1,
    filters: []
  }
}
```

### 5.4 ID 标签

```ts
{
  tagValue: 'VIP',
  userIds: ['u1', 'u2', 'u3']
}
```

### 5.5 SQL 标签

```ts
{
  tagValue: '高价值用户',
  sql: 'SELECT ...',
  userIdField: 'user_id'
}
```

## 6. 构建接口契约

### 6.1 API 文件中的定义

`api/userTag.ts` 中定义了一个构建请求类型：

```ts
export interface BuildUserTagRequest {
  projectAlias: string;
  tagCode: string;
  tagType: number;
  conditionTagRules?: ConditionTagRule[];
  firstOrLastTagRule?: FirstOrLastTagRule;
  eventMetricValueTagRule?: EventMetricValueTagRule;
}
```

但要注意，列表页真正发起构建时，并没有走这个封装。

### 6.2 列表页的实际构建请求

`views/game-users/user-tags/index.vue` 中实际调用的是：

```ts
http.post('/build_user_tag', payload)
```

并且 payload 使用的是 snake_case 风格：

```ts
{
  project_alias,
  tag_code,
  tag_type,
  condition_tag_rules?,
  first_or_last_tag_rule?,
  event_metric_value_tag_rule?
}
```

这说明：

- “保存标签”接口和“构建标签”接口的风格并不完全一致
- 构建接口更接近旧后端原始协议
- `userTagAPI.buildUserTag()` 存在，但页面层并未真正统一使用

## 7. 类型编号映射

列表页构建前会把 `TagType` 转成数字：

```ts
condition    -> 1
first_last   -> 2
metric_value -> 3
id           -> 4
sql          -> 5
```

当前列表页手动构建逻辑只重点支持：

- `1 condition`
- `2 first_last`
- `3 metric_value`

对于 `id` 和 `sql`，旧实现有定义，但未纳入主构建链路。

## 8. 构建后的用户数回写

构建成功后，旧前端会尝试从响应中读取 `user_count`：

- `response.data.user_count`
- `response.data.data.user_count`

如果读取成功，会继续调用：

- `PUT /user-tags/{id}/user-count`

用于回写标签命中用户数。

因此“构建”在旧系统中实际是两段式：

1. 触发构建服务计算标签
2. 回写标签表中的 `userCount`

## 9. 元数据接口

`userTagAPI` 还封装了规则编辑器所需的元数据接口：

- `getProjectEvents(projectAlias)`
- `getEventFields(projectAlias, eventId?)`
- `getUserFields(projectAlias)`

这些接口用于支撑：

- 事件下拉
- 事件属性选择
- 用户属性选择

从架构上看，标签规则编辑本质上依赖项目元数据服务。

## 10. 迁移时的协议注意事项

后续若将该模块迁移到新前端，建议优先处理以下协议问题：

1. 统一保存接口与构建接口的字段命名风格。
2. 收敛 `tagRule` 内部的 camelCase 和 snake_case 混用。
3. 明确 `first_last` 的时间过滤到底由组件层还是转换层负责注入。
4. 决定是否保留 `userTagAPI.buildUserTag()`，避免 API 封装与页面直调并存。
