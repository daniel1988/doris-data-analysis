# 用户标签 - 标签类型与规则编辑器设计

本文档聚焦用户标签模块中的规则编辑器设计，说明不同标签类型的用途、界面组成、输出结构以及与分析组件体系的复用关系。

## 1. 类型划分

`old_frontend/api/userTag.ts` 中定义的标签类型如下：

```ts
export enum TagType {
  CONDITION = 'condition',
  FIRST_LAST = 'first_last',
  METRIC_VALUE = 'metric_value',
  ID = 'id',
  SQL = 'sql'
}
```

创建页 `CreateUserTag.vue` 通过 `formData.tagType` 决定挂载哪个规则编辑器。

## 2. 创建页中的规则编辑器挂载方式

第三步“配置规则”使用条件渲染挂载不同组件：

- `ConditionTagRuleEditor`
- `FirstLastTagRule`
- `MetricValueTagRule`
- `IdTagRule`
- `SqlTagRule`

这些组件都遵循相同的集成方式：

- 通过 `v-model` 输出 `tagRule`
- 接收 `projectAlias`
- 通过 `ref` 暴露 `validate()`

这使得创建页不需要理解每类规则的内部细节，只负责：

- 选择组件
- 读取值
- 校验值
- 在保存时统一序列化

## 3. 条件标签

### 3.1 组件入口

- `old_frontend/views/game-users/components/tag-condition/ConditionTagRuleEditor.vue`

### 3.2 功能定位

条件标签用于根据“事件指标条件”和“用户属性条件”的组合结果给用户赋予某个标签值。

和“一个标签对应一个条件”不同，条件标签支持：

- 一个标签下配置多个标签值
- 每个标签值分别配置命中规则

因此它更接近“枚举型标签”或“分段标签”。

### 3.3 核心界面结构

条件标签编辑器大体分为三部分：

1. 标签值管理
2. 当前标签值下的条件配置
3. 自动更新设置

其中“当前标签值下的条件配置”又拆成两类条件组：

- 事件指标条件组
- 用户属性条件组

并支持三层逻辑关系：

- 事件条件组内的逻辑关系
- 用户属性条件组内的逻辑关系
- 事件组与用户组之间的逻辑关系

### 3.4 规则结构特征

条件标签最终输出为数组，每一项代表一个标签值规则。每个规则包含：

- `tag_value`
- `set_operate`
- `event_filter_group`
- `user_filter_group`

其中：

- `set_operate = 1` 表示交集
- `set_operate = 2` 表示并集

### 3.5 复用关系

条件标签明显复用了旧分群模块的建模思路，尤其是：

- 事件条件
- 用户属性条件
- 过滤器组合逻辑
- 动态时间过滤

相关参考资料：

- `old_frontend/views/game-users/user-groups/README.md`
- `old_frontend/views/game-users/user-groups/condition-group-dev.md`

可以把条件标签理解为“为标签场景做过裁剪的条件分群编辑器”。

## 4. 首末次标签

### 4.1 组件入口

- `old_frontend/views/game-users/components/tag-first-last/FirstLastTagRule.vue`

### 4.2 功能定位

首末次标签用于在指定时间范围内，取用户第一次或最后一次完成某事件时的某个属性值，作为标签值。

典型示例：

- 用户首次登录设备
- 用户最后一次登录区服
- 用户首次付费渠道

### 4.3 核心界面结构

首末次标签主要由三块组成：

1. 指标配置
2. 时间范围配置
3. 更新设置

指标配置中，用户要依次选择：

- 首次还是末次
- 目标事件
- 作为标签值的事件属性
- 事件过滤条件

### 4.4 时间过滤特点

该组件内部维护 `timeFilter`，支持：

- 过去 N 天
- 最近 N 天
- 自某日以来
- 固定时间范围

保存时需要把时间配置注入到 `event_metric.filter_group.filters` 中，对应事件时间字段过滤。

### 4.5 序列化特点

首末次标签是创建页里最特殊的一类，因为组件内部更偏向 camelCase，而保存接口更偏向 snake_case。

`CreateUserTag.vue` 中专门通过 `toSnakeFirstLastRule()` 做转换，将：

- `eventMetric`
- `filterGroup`

转换为：

- `first_or_last_tag_rule.event_metric`
- `filter_group`

这意味着后续若重构该类型，需要优先统一规则结构，减少创建页里的“补丁式转换逻辑”。

## 5. 指标值标签

### 5.1 组件入口

- `old_frontend/views/game-users/components/tag-metric-value/MetricValueTagRule.vue`

### 5.2 功能定位

指标值标签用于在给定时间范围内，统计用户某个事件指标的聚合值，并将该值作为标签值。

常见场景：

- 用户最近 30 天付费总额
- 用户访问次数
- 用户平均在线时长

### 5.3 核心界面结构

该编辑器主要由三部分组成：

1. 分析时段
2. 指标配置
3. 更新设置

指标配置区允许选择：

- 目标事件
- 指标字段
- 聚合方式
- 事件过滤条件

### 5.4 规则结构特征

该类型输出结构以 `event_metric` 为核心，主要包含：

- `event_metric.e_event_id`
- `event_metric.metric.column`
- `event_metric.metric.formula`
- `event_metric.metric.format`
- `event_metric.filter_group`
- `global_filter_group`

其中：

- `filter_group` 更偏向指标级过滤
- `global_filter_group` 作为外层保留字段，目前旧代码中存在但使用感较弱

### 5.5 默认值设计

组件中通过 `createDefaultRule()` 和 `buildSafeRule()` 给规则兜底，默认聚合字段是：

- `e_openid`

默认聚合方式偏向“总次数”。

这一设计的目的是避免新建时出现结构不完整、导致保存或回显失败。

## 6. ID 标签

### 6.1 组件入口

- `old_frontend/views/game-users/components/tag-id/IdTagRule.vue`

### 6.2 功能定位

ID 标签通过直接导入用户 ID 列表给用户打标签，适用于：

- 白名单用户
- 特定运营名单
- 测试用户集合

### 6.3 核心能力

该组件支持：

- 手工输入用户 ID
- 文本文件上传
- 自动去重
- 预览前 20 个用户 ID

### 6.4 当前状态

虽然组件能力完整，但创建页类型卡片没有默认开放该能力，因此更适合作为预留能力记录在案。

## 7. SQL 标签

### 7.1 组件入口

- `old_frontend/views/game-users/components/tag-sql/SqlTagRule.vue`

### 7.2 功能定位

SQL 标签允许开发者直接编写 SQL 查询，查询出用户列表后打上标签，适合：

- 复杂条件组合
- 临时运营圈选
- 需要直接利用数仓能力的场景

### 7.3 核心能力

组件支持：

- 输入 SQL
- 简单格式化
- 基础语法校验
- 内置模板示例
- 配置用户 ID 字段名

### 7.4 当前状态

与 ID 标签类似，SQL 标签在旧前端实现中更像“保留的高级能力”，未在常规新建流程中公开展示。

## 8. 规则编辑器的共同设计模式

### 8.1 统一接入方式

各规则编辑器都尽量遵循：

- 输入 `modelValue`
- 输出 `update:modelValue`
- 暴露 `validate()`

这是典型的“业务容器 + 规则子编辑器”模式。

### 8.2 复用分析组件

用户标签规则配置并没有重新发明一套组件，而是大量复用分析模块组件：

- 属性选择器
- 过滤器行
- 时间选择器
- 指标属性选择器

因此如果要在新前端复刻用户标签，最优路径不是照抄页面，而是优先复用分析模块的原子组件和过滤器协议。

### 8.3 校验边界

创建页只做“当前步骤可否继续”的外层校验，真正的规则字段校验尽量留给子编辑器处理。

这种拆分有两个好处：

- 创建页保持轻量
- 新增标签类型时，只需增加一个规则组件和少量挂载代码

## 9. 重构建议

如果未来重构该模块，建议重点收敛以下问题：

1. 统一 `tagRule` 的命名风格，减少 camelCase 与 snake_case 混用。
2. 将“自动更新设置”抽成共享片段，避免在多个规则编辑器重复实现。
3. 将“时间过滤注入逻辑”从页面层移入规则层或 mapper 层。
4. 明确 `id` 和 `sql` 类型是否正式开放，避免 UI 和能力定义长期不一致。
