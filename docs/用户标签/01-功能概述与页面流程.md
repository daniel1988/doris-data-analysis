# 用户标签 - 功能概述与页面流程

本文档为用户标签模块实现，梳理该模块的页面目标、入口关系、主流程以及核心代码位置，便于后续在新前端中迁移或重构。

## 1. 模块目标

用户标签模块用于为项目内的用户打上可计算、可复用的业务标签。旧前端中主要支持以下能力：

- 标签列表管理
- 创建、编辑、复制、删除标签
- 标签状态启停
- 手动触发标签构建
- 查看标签命中的用户数，并下钻到用户列表

从产品视角看，用户标签是“用户分层”和“用户分析”之间的桥梁：

- 标签可以由事件行为、用户属性、时间范围等规则计算得出
- 构建完成后，标签可以反向作为筛选条件参与后续分析

## 2. 页面入口

### 2.1 路由入口

用户标签路由定义在：

- `old_frontend/router/modules/gameUsers.ts`

其中与用户标签相关的 3 个核心路由为：

- `/game-users/user-tags`
  - 标签列表页
- `/game-users/user-tags/create`
  - 新建标签页
- `/game-users/user-tags/edit/:id`
  - 编辑标签页

路由 `meta.permissions` 体现了该模块的权限拆分：

- `user_tag.view`
- `user_tag.create`
- `user_tag.edit`

### 2.2 页面文件

页面代码主要分布在：

- `old_frontend/views/game-users/user-tags/index.vue`
- `old_frontend/views/game-users/user-tags/CreateUserTag.vue`

## 3. 页面职责划分

### 3.1 标签列表页

列表页负责日常运维动作，主要包括：

- 按项目、标签类型、状态、关键字检索标签
- 展示标签名称、编码、类型、更新方式、用户数、创建人、创建时间
- 执行编辑、构建、启停、复制、删除等操作
- 用户数可点击，下钻打开用户明细抽屉

列表页不承担复杂规则编辑职责，它只负责：

- 查询标签数据
- 将标签规则交给构建逻辑转换为构建请求

### 3.2 创建/编辑页

创建页采用四步流程：

1. 基础信息
2. 选择类型
3. 配置规则
4. 确认保存

创建页统一承担以下职责：

- 收集标签基础信息
- 根据标签类型挂载对应规则编辑器
- 对规则进行校验
- 保存时序列化为后端接口需要的 `tagRule`
- 编辑模式下回显已有标签数据

## 4. 页面主流程

### 4.1 列表页主流程

列表页的典型访问流程如下：

1. 进入 `/game-users/user-tags`
2. 从 `localStorage.currentProjectAlias` 同步当前项目别名
3. 调用 `userTagAPI.getUserTags()` 拉取分页列表
4. 展示表格与分页器
5. 用户可继续执行查询、编辑、构建、启停、删除等动作

该页有一个重要约定：

- 当前项目别名不是在页内手动选择，而是和顶部项目切换保持同步

### 4.2 创建页主流程

新建标签时的主流程如下：

1. 从当前项目上下文自动填充 `projectAlias`
2. 自动生成 `tagCode`
3. 默认将 `tagShowName` 与 `tagCode` 同步
4. 进入“选择类型”步骤
5. 挂载对应规则编辑器录入 `tagRule`
6. 确认信息后调用创建接口保存
7. 保存成功后返回标签列表

### 4.3 编辑页主流程

编辑标签时的主流程如下：

1. 根据路由参数 `id` 调用 `userTagAPI.getUserTagById(id)`
2. 将返回结果回填到 `formData`
3. 若已有完整 `tagType + tagRule`，直接跳转到确认步骤
4. 用户修改后调用更新接口保存

### 4.4 复制标签主流程

复制不是独立页面，而是复用创建页：

1. 列表页点击“复制标签”
2. 跳转到 `/game-users/user-tags/create?copy={id}`
3. 读取原标签详情
4. 自动生成新的 `tagCode` 与 `tagShowName`
5. 用户确认后按“新建标签”逻辑保存

## 5. 标签类型概览

旧前端在类型枚举中定义了 5 类标签：

- `condition`
- `first_last`
- `metric_value`
- `id`
- `sql`

但从创建页的类型卡片实现看，当前真正面向常规用户开放的主要是前 3 类：

- 条件标签
- 首末次标签
- 指标值标签

`id` 和 `sql` 虽然保留了规则组件，但在创建页的类型选择区被注释掉了，更像是预留能力或实验能力。

## 6. 核心代码定位

### 6.1 页面与路由

- `old_frontend/router/modules/gameUsers.ts`
- `old_frontend/views/game-users/user-tags/index.vue`
- `old_frontend/views/game-users/user-tags/CreateUserTag.vue`

### 6.2 API 封装

- `old_frontend/api/userTag.ts`

### 6.3 规则编辑器

- `old_frontend/views/game-users/components/tag-condition/ConditionTagRuleEditor.vue`
- `old_frontend/views/game-users/components/tag-first-last/FirstLastTagRule.vue`
- `old_frontend/views/game-users/components/tag-metric-value/MetricValueTagRule.vue`
- `old_frontend/views/game-users/components/tag-id/IdTagRule.vue`
- `old_frontend/views/game-users/components/tag-sql/SqlTagRule.vue`

### 6.4 复用组件

用户标签大量复用了已有的分析组件能力，例如：

- `PropertySelect`
- `BaseFilterItem`
- `MetricPropertyPicker`
- `DynamicStaticDatePicker`
- `UserDrilldownDrawer`

这说明用户标签模块并不是孤立页面，而是建立在分析组件体系上的业务组合层。

## 7. 推荐阅读顺序

如果后续要继续迁移或重构，建议按以下顺序阅读旧代码：

1. `router/modules/gameUsers.ts`
2. `views/game-users/user-tags/index.vue`
3. `views/game-users/user-tags/CreateUserTag.vue`
4. `api/userTag.ts`
5. 各标签类型对应规则编辑器

先从页面流程理解，再进入规则结构，会比直接读规则组件更容易建立全局认知。
