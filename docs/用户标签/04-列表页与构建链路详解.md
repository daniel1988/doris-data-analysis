# 用户标签 - 列表页与构建链路详解

本文档重点说明  用户标签列表页的职责、操作流转以及“手动构建标签”的完整链路。这部分是用户标签模块里最容易在迁移时出错的区域。

## 1. 列表页职责

列表页实现位于：

- `old_frontend/views/game-users/user-tags/index.vue`

它承担的职责包括：

- 拉取用户标签分页列表
- 根据条件过滤列表
- 跳转创建与编辑页
- 调用启停、删除、复制等管理操作
- 手动触发标签构建
- 维护并展示标签命中用户数
- 打开用户下钻抽屉

换句话说，列表页既是“标签管理台”，也是“标签执行台”。

## 2. 列表查询流程

### 2.1 查询条件

列表页维护的查询条件为：

- `projectAlias`
- `tagType`
- `status`
- `keyword`
- `page`
- `pageSize`

其中 `projectAlias` 不是本页独立选择，而是从顶部项目上下文同步得到。

### 2.2 当前项目同步

页面通过 `syncProjectAliasFromLocal()` 从：

- `localStorage.currentProjectAlias`

同步当前项目别名，并更新：

- `currentProjectAlias`
- `queryParams.projectAlias`

页面还监听 `storage` 事件，以便项目切换后自动刷新列表。

### 2.3 拉取列表

列表加载时调用：

```ts
userTagAPI.getUserTags({
  ...queryParams,
  page: pagination.page,
  pageSize: pagination.pageSize
})
```

返回后更新：

- `tableData`
- `pagination.total`

## 3. 列表操作流转

### 3.1 新建

点击“创建标签”后：

```ts
router.push('/game-users/user-tags/create')
```

### 3.2 编辑

点击“编辑”后：

```ts
router.push(`/game-users/user-tags/edit/${row.id}`)
```

### 3.3 复制

点击“复制标签”后：

```ts
router.push(`/game-users/user-tags/create?copy=${row.id}`)
```

复制本质上是带参数进入创建页。

### 3.4 启停

点击“启用/禁用”后：

1. 弹出确认框
2. 调用 `userTagAPI.updateTagStatus(row.id, newStatus)`
3. 刷新列表

### 3.5 删除

点击“删除”后：

1. 弹出确认框
2. 调用 `userTagAPI.deleteUserTag(row.id)`
3. 刷新列表

## 4. 手动构建的整体流程

“构建”是列表页中最重要的业务操作。其核心流程如下：

1. 用户点击“构建”
2. 页面校验该标签是否存在 `tagRule`
3. 根据 `tagType` 转成数字枚举
4. 将 `row.tagRule` 组装为构建接口需要的 payload
5. 调用 `/build_user_tag`
6. 从响应中读取 `user_count`
7. 若存在 `user_count`，继续调用更新用户数接口
8. 刷新列表

## 5. 类型编号转换

构建前，列表页通过 `getTagTypeNumber()` 做如下映射：

- `condition` -> `1`
- `first_last` -> `2`
- `metric_value` -> `3`
- `id` -> `4`
- `sql` -> `5`

但旧实现当前只显式支持前 3 类进入主构建分支。

## 6. 构建 payload 组装规则

构建请求公共字段如下：

```ts
{
  project_alias: row.projectAlias,
  tag_code: row.tagCode,
  tag_type: tagTypeNum
}
```

在此基础上，根据标签类型补充不同字段。

### 6.1 条件标签构建

条件标签使用：

```ts
payload.condition_tag_rules = Array.isArray(rule) ? rule : [rule]
```

这里有一个兼容性处理：

- 如果 `tagRule` 不是数组，也会被包成单元素数组

说明后端期望的是“多个标签值规则”的数组结构。

### 6.2 首末次标签构建

首末次标签是构建链路里最复杂的一类，旧代码做了两层兼容：

1. 优先读取已经序列化好的 snake_case 结构
2. 如果只有 camelCase 结构，则在列表页临时转换

优先分支：

```ts
const snake = row.tagRule?.first_or_last_tag_rule;
if (snake?.event_metric) {
  payload.first_or_last_tag_rule = snake;
}
```

回退分支会从：

- `row.tagRule.firstOrLastTagRule`
- 或 `row.tagRule`

中提取 `eventMetric`，再组装成：

```ts
{
  event_metric: {
    e_event_id,
    type,
    metric,
    filter_group
  }
}
```

#### 6.2.1 时间过滤注入

若 `row.tagRule.timeFilter` 存在，列表页会额外把时间过滤转换后注入 `filter_group.filters`。

支持的转换规则包括：

- `past_days`
- `recent_days`
- `since_date`
- `fixed_range`

其中动态日期过滤会使用：

- `column.field = 'e_event_time'`
- `operator = OP.DYNAMIC_DATES`

这说明时间过滤不是天然固化在规则对象中，而是有一部分依赖构建阶段再加工。

### 6.3 指标值标签构建

指标值标签直接取：

```ts
payload.event_metric_value_tag_rule =
  row.tagRule?.eventMetricValueTagRule || row.tagRule
```

和首末次相比，这类规则的构建转换更直接，说明它的保存结构与构建结构更接近。

## 7. 构建接口调用特点

旧前端构建时直接调用：

```ts
http.post('/build_user_tag', payload)
```

注意点：

- 没有直接复用 `userTagAPI.buildUserTag()`
- 接口路径不是 REST 风格的 `/user-tags/build`
- 请求字段使用 snake_case

这部分体现出旧系统的历史兼容痕迹，后续若统一 API 层，这里是优先重构点。

## 8. 构建结果处理

构建完成后，页面会尝试从响应中读取用户数：

```ts
response.data.user_count
response.data.data.user_count
```

读到后再调用：

```ts
userTagAPI.updateUserCount(row.id, userCount)
```

然后：

- 更新当前行的 `row.userCount`
- 刷新列表

如果回写用户数失败，页面仍然提示构建成功，只是打印警告。

这说明旧实现将“构建成功”和“统计值回写成功”视为两个不同层级的结果。

## 9. 用户数下钻链路

表格中的 `userCount` 支持点击查看用户明细，但有前置条件：

- `row.userCount > 0`

点击后会：

1. 设置 `selectedTag = row`
2. 打开 `drawerVisible`
3. 将标签信息传给 `UserDrilldownDrawer`

传入参数包含：

- `id`
- `tagCode`
- `tagShowName`
- `userCount`
- `projectAlias`

因此列表页不直接渲染用户明细，而是通过通用下钻抽屉完成复用。

## 10. 列表页的设计特点

从代码组织上看，列表页有几个明显特征：

### 10.1 页面内业务逻辑偏重

虽然有 `userTagAPI`，但构建逻辑、类型转换逻辑、规则兼容逻辑仍然堆在页面中，导致页面承担了较多业务拼装责任。

### 10.2 页面层做了大量兼容

典型包括：

- 标签规则数组与对象兼容
- 首末次规则 camelCase 与 snake_case 兼容
- 构建响应 `user_count` 嵌套层级兼容

### 10.3 强依赖历史字段协议

页面直接依赖：

- `project_alias`
- `tag_code`
- `tag_type`
- `condition_tag_rules`
- `first_or_last_tag_rule`
- `event_metric_value_tag_rule`

这种强耦合适合写清楚文档，不适合继续在页面里扩散。

## 11. 迁移建议

若后续在新前端中重做列表页，建议按以下方向收敛：

1. 将构建 payload 组装抽到单独 mapper 文件。
2. 将 `TagType -> tagTypeNum` 映射沉淀为共享常量。
3. 将首末次时间过滤注入逻辑移出页面。
4. 统一使用 API 层，不再让页面混用 `userTagAPI` 与 `http.post()`。
5. 对构建响应定义稳定的类型，去掉多层兜底判断。

这样才能让“列表展示”和“构建执行”重新职责分离。
