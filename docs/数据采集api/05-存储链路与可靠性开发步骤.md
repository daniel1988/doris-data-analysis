# 05-存储链路与可靠性开发步骤

## 1. 存储策略

建议按阶段推进：

### 阶段 1（直写）
- Service 直接写入日志
- 优点：简单、可快速验证链路
- 缺点：高峰抗压能力弱

### 阶段 2（分钟文件，推荐）
- 原始采集事件按“分钟”写入本地文件（JSON Lines）
- 每分钟触发一次解析任务，读取对应分钟文件并通过 Doris Stream Load 导入
- 采集 API 仅保证“成功落本地分钟日志”，Doris 写入异步完成

建议文件命名：
- `logs/collect/YYYYMMDD/HH/mm.log`
- 例如：`logs/collect/20260323/10/42.log`

建议单行格式（JSONL）：
- 每行一个标准化后的 `NormalizedEvent`
- 强制包含：`project_alias`、`openid`、`event_id`、`event_time`、`ingest_time`、`trace_id`

## 1.1 分钟解析写 Doris 流程

1. 定时任务每分钟触发（如 `10:43:05` 处理 `10:42.log`）
2. 读取分钟文件并逐行反序列化
3. 执行行级校验与脏数据隔离
4. 组装 Stream Load 请求（按项目或批次）
5. 发送到 Doris FE 的 Stream Load 接口
6. 校验返回结果（Status/NumberTotalRows/NumberLoadedRows）
7. 成功后写入 checkpoint（防重复）
8. 将已完成文件移动到 `archive/`，失败文件移动到 `failed/`

dc_api 推荐入库流程（参考 data-collect-api 项目）：
- 先按 `project_alias` 对分钟事件分组
- 每个项目单独执行一次 Stream Load，目标库为 `{db_prefix}{project_alias}`
- 每次请求使用独立 `label`，便于幂等控制与失败重放
- 回放成功后写入本地 checkpoint（如 `logs/collect/_checkpoint/replay_checkpoint.json`）

## 1.2 Stream Load 导入约定

推荐使用参数：
- `format: json`
- `read_json_by_line: true`
- `strip_outer_array: false`
- `strict_mode: false`（MVP 可用，生产按数据质量策略调整）
- `columns: ...`（基于待导入行的 key 动态拼接显式列映射）
- `label: collect_{project}_{yyyyMMddHHmm}`（用于幂等与去重）
5. 写入成功后记录 checkpoint（防重复）
返回值校验建议：
- `Status` 必须为 `Success` 或 `Publish Timeout`（结合业务是否视为成功）
- `NumberLoadedRows` 与预期行数差异超过阈值时标记告警
- 保留 `ErrorURL` 便于排查脏数据
- 当 `NumberTotalRows > 0` 且 `NumberLoadedRows = 0` 时按失败处理

## 1.3 event_data 字段校验与缺失字段补齐

为避免上报字段不完整导致 Doris 导入失败，写入前需要基于 `event_data` 表结构做字段校验与补齐：

1. 读取并缓存 `event_data` 目标表字段清单（建议按 `project_alias` 缓存，定时刷新）
2. 对每条待导入记录执行字段映射校验：
   - 上报中存在但不在 `event_data` 表中的字段：按策略丢弃并记录告警
   - `event_data` 表中存在但上报缺失的字段：补齐默认值或 `NULL`
3. 生成最终 Stream Load 的 `columns` 映射，确保列顺序与目标表一致

缺失字段补齐建议：
- 公共核心字段（如 `e_event_id`, `e_event_time`, `e_openid`）：缺失时判定为非法数据，进入 `failed/`
- 非核心业务字段：允许补 `NULL` 或配置默认值
- 动态属性字段：从 `properties` 解析后映射到 `event_data` 的 `e_` 前缀列（如 `order_id -> e_order_id`）

properties 字段解析规则：
- `properties` 的 key 写入前统一转换为 `e_{key}`
- 已带 `e_` 前缀的 key 保持不变
- 非法 key（空值、非法字符）直接丢弃并记录告警
- 解析后列与 `event_data` 实际字段做交集校验，不存在列按策略处理（丢弃或入失败文件）
- 未知列策略建议可配置：`append/drop/fail`（默认 `append`）

字段校验产物建议落日志：
- `unknown_fields`：上报但目标表不存在
- `missing_required_fields`：目标表必填但上报缺失
- `filled_default_fields`：自动补齐的字段清单

## 2. Repo 抽象

建议定义接口：
- `EventSink`：`Push(events []NormalizedEvent) error`
- `FailStore`：`Save(events []NormalizedEvent, reason string) error`

实现类：
- `MQSink`
- `DorisSink`（调试或小流量）
- `MySQLFailStore` / `FileFailStore`
- `MinuteFileSink`（新增，负责按分钟落地本地日志）
- `MinuteFileParser`（新增，负责分钟文件解析并写 Doris）
- `DorisStreamLoadSink`（新增，负责调用 Stream Load 导入）
- `EventSchemaRepo`（新增，负责读取/缓存 `event_data` 字段定义）

## 2.1 本地文件写入接口建议

- `MinuteFileSink.Append(event NormalizedEvent) error`
- `MinuteFileSink.AppendBatch(events []NormalizedEvent) error`
- `MinuteFileSink.Flush(minute string) error`

实现要求：
- 并发安全（同一分钟多协程写入）
- 落盘可控（行缓冲 + 定时 flush）
- 文件损坏保护（写入失败时记录 error log）

## 3. 失败处理机制

- 下游不可用：重试 + 指数退避
- 多次失败：写死信（DLQ）
- 保留失败事件与错误原因，支持补偿回放

补充到“分钟文件方案”：
- 分钟解析写 Doris 失败时，原文件不删除，保留在 `failed/`
- 失败记录包含：分钟文件名、失败行号、错误原因、重试次数
- 支持按分钟文件重放（人工或定时重试）
- Stream Load 返回 `Label Already Exists` 时，按幂等成功处理并记录去重日志

## 4. 限流与熔断

- 按 `project_alias` 做租户级限流
- 达到阈值后返回 `429` 与重试建议
- 下游异常时启用短路保护，防止雪崩

补充：
- 当分钟文件写入磁盘失败时，立即触发采集降级告警
- 当 `failed/` 累积超过阈值时，暂停新解析任务，避免持续放大错误

## 5. 本步骤验收标准

- 峰值流量下接口可稳定返回
- 下游故障不会导致 API 进程崩溃
- 失败事件可查询、可回放
- 本地分钟日志可稳定生成，文件路径与命名符合规范
- 分钟解析任务可通过 Stream Load 将指定分钟文件导入 Doris 且具备防重能力
- 导入前可完成 `event_data` 字段校验，并对缺失字段执行补齐策略
