# 02-接口与数据模型开发步骤

## 1. 输入格式映射

根据 [数据格式.md](file:///d:/gitee/dmp_admin_v2/docs/数据采集api/数据格式.md) 定义三类 DTO：

1. `SingleEventRequest`
2. `SingleUserBatchRequest`
3. `MultiUserBatchRequest`（数组）

公共事件对象建议抽象为：
- `event_id`（string）
- `event_name`（string）
- `event_time`（int64，秒级时间戳）
- `properties`（map[string]any）

公共用户上下文建议抽象为：
- `openid`（string）
- `channel_id`（string）
- `device_id`（string）

## 2. 接口契约定义

建议固定三个入口，便于权限和限流策略分离：

- `POST /api/v1/collect/event`（单事件）
- `POST /api/v1/collect/user-batch`（单用户多事件）
- `POST /api/v1/collect/batch`（多用户多事件）

统一响应：

```json
{
  "code": 200,
  "data": {},
  "message": "success"
}
```

## 3. 参数校验规则

### 3.1 必填校验
- `openid`、`event_id`、`event_time` 必填
- 批量模式下 `events` 不可为空

### 3.2 数据约束
- `event_time` 允许时间窗口：`now-7d ~ now+5m`
- `properties` 单事件键数量上限（如 100）
- 字段名与字符串值长度上限（防止超长脏数据）

### 3.3 去重与幂等字段
- 推荐支持 `event_uuid`（可选）
- 无 `event_uuid` 时使用 `(openid,event_id,event_time,device_id)` 作为近似幂等键

## 4. 标准化输出对象

在 API 层不要直接向后续链路传原始结构，统一转换为 `NormalizedEvent`：
- `project_alias`
- `openid`
- `channel_id`
- `device_id`
- `event_id`
- `event_name`
- `event_time`
- `event_properties_json`
- `receive_time`
- `source_type`（sdk/server）

## 5. 本步骤验收标准

- 三类请求体均可正确绑定
- 非法数据返回统一错误结构与明确 message
- 合法数据可转换为统一 `NormalizedEvent` 集合
